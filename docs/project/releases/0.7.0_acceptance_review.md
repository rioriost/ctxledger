# `0.7.0` Acceptance Review

## Purpose

This document records a release-facing acceptance assessment for the bounded
`0.7.0` task-recall slice implemented in `ctxledger`.

It is intended to answer a practical release question:

> Does the currently implemented `0.7.0` task-recall slice satisfy the
> documented closeout expectations strongly enough to be treated as accepted for
> its intended bounded scope?

This is **not** a new architecture proposal.

It is a release-facing acceptance review that evaluates the current repository
reading against the existing `0.7.0` plan, roadmap/changelog status notes,
implementation-status review, and operator-facing behavior summary.

---

## Acceptance scope

This review is specifically about the bounded `0.7.0` task-recall slice
described in:

- `docs/project/releases/plans/versioned/task_recall_0_7_0_plan.md`
- `docs/project/product/roadmap.md`
- `docs/project/releases/CHANGELOG.md`
- `docs/project/releases/0.7.0_status_review.md`
- `docs/project/releases/0.7.0_behavior_summary.md`

This review does **not** claim that all longer-term task-thread, return-target,
or concept-routing ambitions are complete.

It assesses whether the repository has completed the **bounded implemented
`0.7.0` slice** strongly enough to accept it as a release-worthy milestone
outcome.

---

## Review result

**Acceptance result:** accepted  
**Acceptance reading:** accepted as a **bounded `0.7.0` milestone slice**  
**Review status:** passed with explicit bounded-scope caveats

The current repository reading supports the following conclusion:

- the `0.7.0` bounded task-recall slice is implemented
- the slice is documented clearly enough to support a release-facing reading
- the slice is validated broadly enough for its intended bounded scope
- the slice preserves canonical workflow/checkpoint truth while adding richer
  derived recall and explanation behavior
- the remaining deferred work is explicit rather than ambiguous

This means the repository can reasonably treat `0.7.0` as **accepted for the
implemented bounded scope**.

---

## Acceptance criteria used

The primary acceptance baseline comes from:

- `docs/project/releases/plans/versioned/task_recall_0_7_0_plan.md`

Especially:

- `§4.1 Current implementation progress`
- `§7 Retrieval Questions 0.7.0 Must Support Better`
- `§16 Implementation phases`
- `§17 Validation Criteria`
- `§19 Deliverables`

The practical acceptance criteria used in this review are:

1. the current repository actually delivers a meaningful bounded task-recall
   improvement rather than only a planning surface
2. workspace-scoped continuation selection is more robust than pure recency
3. the repository can distinguish latest considered work from the selected
   continuation target
4. prior-mainline recovery before a latest detour exists in bounded form
5. `memory_search` includes a real bounded concept-to-task recovery bridge
6. return-target, primary-objective, and task-thread details are surfaced
   clearly enough for the bounded slice
7. the canonical PostgreSQL-first system-of-record boundary remains intact
8. the current docs distinguish:
   - what `0.7.0` implemented
   - what remains intentionally bounded
   - what remains deferred or incomplete
9. the current implementation is backed by focused validation rather than by
   roadmap prose alone
10. the accepted reading remains bounded and does not over-claim a full
    task-thread or return-target architecture

## What is accepted here

This review accepts the following bounded reading of `0.7.0`:

- `ctxledger` now has a substantial task-recall slice that improves
  continuation recovery after interruptions or side-task detours
- the system can now surface richer explanation details for why one workflow was
  selected over another
- the system can now expose bounded prior-mainline recovery and bounded
  latest-detour-versus-selected-target separation
- `memory_search` now has a real bounded concept-to-task support bridge
- the current slice remains derived-on-top-of-canonical-state rather than a new
  canonical workflow-memory model

That is enough to treat `0.7.0` as **accepted for its intended bounded slice**.

---

## What is not accepted here

This acceptance does **not** imply that:

- all originally imagined `0.7.0` ambitions are fully complete
- a full task-thread model already exists
- a first-class persisted return-target entity already exists
- all concept-to-task routing work is complete
- all adjacent retrieval surfaces have already been updated
- maintainability hardening and file-splitting work is complete
- the repository has no remaining post-acceptance cleanup or closeout work

Those items remain valid as either:

- bounded follow-up work
- explicit deferred work
- or release-process hardening work

## Remaining caveats after acceptance

The current accepted reading still leaves some explicit caveats:

1. concept-to-task recovery remains bounded rather than broad
2. broader retrieval-surface integration remains a scoping question
3. return-target and task-thread fields remain derived explanation surfaces
4. maintainability hardening for oversized task-recall-related files is now
   complete for the bounded `0.7.0` split stream:
   - the first oversized runtime task-recall source files have now been split
   - the first oversized task-recall test files have now been split
   - the oversized memory-context visibility test file has now also been split
   - the oversized memory-context grouping test file has now also been split
   - the oversized memory-context task-recall test file has now also been split
   - the final oversized memory-context grouping-summary-routes test file has
     now also been split into:
     - `tests/memory/test_service_context_details_grouping_summary_routes_multiflow.py`
     - `tests/memory/test_service_context_details_grouping_summary_routes_ordering.py`
5. bounded acceptance should not be confused with a full milestone-closure claim
   for every longer-term task-recall ambition

These caveats are acceptable because they are now explicit.
