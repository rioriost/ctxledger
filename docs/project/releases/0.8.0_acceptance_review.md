# `0.8.0` Acceptance Review

## Purpose

This document records a release-facing acceptance assessment for the bounded
`0.8.0` remember-path slice implemented in `ctxledger`.

It is intended to answer a practical release question:

> Does the current repository satisfy the documented `0.8.0` remember-path
> acceptance shape strongly enough to be treated as accepted for its intended
> bounded scope?

This review is not a new architecture proposal.

It evaluates the current repository reading against the documented `0.8.0`
remember-path direction, current product docs, operator runbooks, and current
implementation/testing posture.

---

## Acceptance scope

This review is specifically about the bounded `0.8.0` remember-path slice
described by the repository’s current implementation and supporting docs,
including:

- `docs/project/releases/plans/versioned/remember_path_0_8_0_plan.md`
- `docs/project/product/roadmap.md`
- `docs/project/product/specification.md`
- `docs/project/product/mcp-api.md`
- `docs/memory/design/memory_get_context_service_contract.md`
- `docs/memory/runbooks/summary_build_runbook.md`
- `docs/operations/runbooks/small_auth_operator_runbook.md`

This review does **not** claim that every longer-term memory, graph, or ranking
ambition associated with future milestones is complete.

It assesses whether the repository now satisfies the bounded `0.8.0`
implementation shape strongly enough to be accepted as a release-worthy
remember-path milestone.

---

## Review result

**Acceptance result:** accepted with bounded-scope caveats  
**Acceptance reading:** accepted as a **bounded `0.8.0` remember-path milestone slice**  
**Review status:** passed with explicit follow-up candidates preserved

The current repository supports the following bounded reading:

- remember-path behavior is materially stronger than earlier sparse
  episode-note-only operation
- canonical memory accumulation is now more visible and more explainable
- checkpoint-origin and completion-origin memory behavior is now surfaced more
  clearly
- `memory_get_context` now exposes a clearer primary grouped surface and an
  explicit primary-only client path
- derived AGE readiness is now operationally legible enough to support bounded
  operator judgment
- local small-stack operation is now better aligned with ordinary development
  posture
- remaining follow-up work is explicit rather than hidden inside ambiguous
  milestone claims

This means the repository can reasonably treat `0.8.0` as **accepted for the
implemented bounded scope**.

---

## Acceptance criteria used

The practical acceptance baseline for this review is:

1. the repository now records richer remember-path artifacts than sparse episode
   storage alone
2. remember-path behavior is distinguishable across checkpoint-origin and
   completion-origin memory
3. `memory_get_context` has a primary grouped reading that is explicit enough
   for client use
4. `primary_only` exists as a bounded response-shaping mode rather than a
   separate retrieval algorithm
5. search behavior gives a clearer bounded preference when completion-origin and
   checkpoint-origin memories compete on tied lexical evidence
6. AGE readiness is clearer than a single ready/unavailable binary and exposes
   bounded remediation guidance
7. operator-facing stats expose defined canonical-summary and derived-graph
   metrics
8. local small-stack guidance is operationally safe enough for normal
   development without forcing external embedding credentials
9. the repository docs describe canonical-versus-derived boundaries consistently
10. the current implementation is backed by focused validation rather than by
    docs alone

---

## Detailed review

## 1. The repository now records richer remember-path artifacts

### Expected

`0.8.0` should strengthen the path from workflow activity to durable memory so
the system does not stop at sparse episodic storage.

### Evidence

The current repository reading now includes:

- checkpoint-origin memory capture through `workflow_checkpoint_auto`
- completion-origin memory capture through `workflow_complete_auto`
- explicit remember-path explainability surfaces
- clearer visibility into memory-item provenance
- stronger operator-facing visibility into summary volume and graph posture

The remember-path surface is therefore no longer just:

- workflow state
- maybe an episode
- maybe a single memory item

It now exposes a more legible bounded multi-layer recording shape.

### Result

**Pass**

The bounded milestone now materially improves memory accumulation quality.

---

## 2. Checkpoint-origin and completion-origin memory are distinguished clearly

### Expected

The system should make it clearer whether useful memory came from checkpoint
promotion or completion-time promotion.

### Evidence

The current repository now consistently distinguishes:

- `workflow_checkpoint_auto`
- `workflow_complete_auto`

This distinction is visible across:

- schema and provenance handling
- `memory_get_context` explainability
- search ranking details
- CLI memory/provenance stats
- tests that cover checkpoint-vs-completion behavior directly

This makes the remember path operationally legible in a way that earlier slices
did not provide.

### Result

**Pass**

The current bounded slice clearly distinguishes checkpoint-origin and
completion-origin memory.

---

## 3. `memory_get_context` now has a clearer primary grouped surface

### Expected

The repository should make it easier for clients to treat grouped output as the
primary surface rather than relying on flatter compatibility projections.

### Evidence

The current repository now explicitly treats:

- `memory_context_groups`

as the primary grouped hierarchy-aware surface.

It also now exposes stronger additive signals around:

- primary grouped reading
- route-aware interpretation
- compatibility-oriented top-level fields
- convenience-oriented fields

The contract documents now consistently describe the grouped path as primary and
flatter fields as compatibility or convenience surfaces.

### Result

**Pass**

The primary grouped reading is now explicit enough to support bounded client
adoption.

---

## 4. `primary_only` is a meaningful bounded response-shaping mode

### Expected

Clients should be able to ask for the primary grouped path without being forced
to depend on flatter compatibility-oriented explainability fields.

### Evidence

The current repository now includes a bounded request-shaping flag:

- `primary_only`

Its current reading is explicitly documented as:

- preserving the same retrieval behavior
- preserving the grouped primary surface
- preserving route/selection interpretation metadata
- omitting flatter compatibility-oriented projections where appropriate

This behavior is now reflected across:

- service request shape
- MCP schema and handler wiring
- serialization behavior
- contract docs
- test coverage for primary-only omission behavior

### Result

**Pass**

`primary_only` now exists as a bounded and explicit client-facing contract shape.

---

## 5. Search ranking now has a bounded completion-vs-checkpoint tie-break rule

### Expected

When lexical evidence ties, the system should expose a clearer bounded rule for
how completion-origin memory and checkpoint-origin memory are ordered.

### Evidence

The current repository now includes a bounded preference rule in which
completion-origin memory can outrank checkpoint-origin memory when lexical
evidence ties.

The search-ranking explanation surface also now records this through an explicit
reason code:

- `workflow_complete_auto_tiebreak`

This does not claim a complete future ranking model.
It does provide a clear bounded rule that is:

- understandable
- explainable
- test-backed

### Result

**Pass**

The current search behavior now includes a bounded and explicit completion-vs-
checkpoint preference rule.

---

## 6. AGE readiness is now operationally more legible

### Expected

The repository should provide more than a simplistic ready/unavailable binary if
operators are expected to reason about graph-backed auxiliary behavior safely.

### Evidence

The current AGE readiness reading now includes bounded readiness fields such as:

- `readiness_state`
- `stale`
- `degraded`
- `operator_action`

The current readiness posture also distinguishes among states such as:

- `ready`
- `stale`
- `degraded`
- `unknown`

This is supported in current implementation and reflected in docs and tests.

The important architectural boundary is also preserved:

- canonical relational summary state remains primary
- derived AGE graph posture remains supplementary and degradable

### Result

**Pass**

AGE readiness is now clear enough to support bounded operator reasoning.

---

## 7. Operator-facing stats now expose defined summary and graph metrics

### Expected

The current milestone should make operators better able to understand whether
the repository is accumulating canonical summary state and what posture the
derived graph layer is in.

### Evidence

The current repository now exposes defined metrics including:

- `memory_summary_count`
- `memory_summary_membership_count`
- `age_summary_graph_ready_count`
- `age_summary_graph_stale_count`
- `age_summary_graph_degraded_count`
- `age_summary_graph_unknown_count`

These appear in current operator-facing stats surfaces and are described as:

- canonical summary volume first
- derived graph posture second

This is a materially stronger operator story than only exposing generic memory
counts.

### Result

**Pass**

The bounded operator-metrics improvement is present.

---

## 8. Local small-stack posture is now safer for normal development

### Expected

for ordinary development usage.

The current repository reading now treats local/dev embedding posture more

- explicit reminder that external providers are opt-in rather than required for

The repository also keeps `.env` auto-loading as the ordinary local operator

### Result

**Pass**

The local operator posture is materially safer and clearer.

---

## 9. Canonical-versus-derived boundaries remain intact

### Expected

The remember-path milestone must not blur canonical truth with derived support

### Evidence

The current repository consistently preserves the following boundaries:

- canonical summary ownership remains relational
- AGE summary graph state remains derived

- `primary_only` changes shaping, not retrieval truth

This is reflected consistently across current docs, runtime behavior, and tests.

### Result

**Pass**

The architectural boundary remains intact.

### Evidence

**Pass**

---

## Explicit bounded-scope caveats

This acceptance review is intentionally bounded.

It does **not** claim completion of:

- a full future graph-operator metrics model with per-cause persisted event
  accounting
- a complete future ranking model across all semantic, task-recall, and
  remember-path signals
- retirement of all compatibility-oriented `memory_get_context` fields
- a broad graph-first retrieval redesign
- a complete future release closeout package beyond the current bounded
  acceptance posture

These remain valid future work items, but they do not block acceptance of the
current bounded `0.8.0` slice.
