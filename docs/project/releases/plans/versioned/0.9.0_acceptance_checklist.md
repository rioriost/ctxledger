# `0.9.0` Acceptance Checklist

## Purpose

This checklist defines the release-facing acceptance criteria for the bounded
`0.9.0` milestone.

The milestone is intended to strengthen `ctxledger` and repository `.rules`
together so the system is better at:

- resumability from minimal user prompts
- bounded historical progress recall
- automatic interaction-memory capture
- failure-pattern reuse and avoidance
- durable file-work metadata capture without indexing Git-managed file contents

This checklist should be used as the primary acceptance gate for the milestone.

It is intentionally practical.
The goal is to make it possible to answer:

> Is `0.9.0` complete enough to accept as a bounded release slice?

---

## Reading rules

When using this checklist:

- treat PostgreSQL-backed workflow, checkpoint, and memory state as canonical
- treat summaries, embeddings, ranking details, and graph-linked structures as
  derived support layers unless explicitly defined otherwise
- treat `.rules` as part of the acceptance surface when it materially affects:
  - resumability
  - historical recall
  - failure avoidance
  - interaction-memory discipline
- do not over-claim broad graph-memory or general conversational QA behavior
  beyond the bounded `0.9.0` scope

---

## Acceptance result values

Each checklist item should be read using one of:

- `pass`
- `partial`
- `fail`
- `deferred`

The milestone should be treated as **accepted** only when:

- all required core items are `pass`
- any `partial` items are explicitly bounded and non-blocking
- deferred items are explicitly outside the bounded `0.9.0` scope

---

## Core acceptance areas

`0.9.0` is accepted only if all of these areas are satisfied strongly enough:

1. minimal-prompt resumability
2. bounded historical progress recall
3. automatic interaction-memory capture
4. failure-pattern reuse and avoidance
5. file-work metadata capture
6. `.rules` and implementation alignment
7. bounded operator-facing observability
8. validation coverage
9. documentation and release-readiness alignment

---

# 1. Minimal-prompt resumability

## 1.1 Resume from minimal user prompts

### Goal

A user can say things like:

- `resume`
- `continue`
- `作業を再開`

and the agent can recover the intended continuation target with materially less
ambiguity than earlier slices.

### Acceptance checks

- [ ] `pass` — the repository has a documented bounded contract for minimal-prompt
      resume behavior
- [ ] `pass` — the resume path can recover, in representative cases:
  - selected workflow
  - latest relevant checkpoint
  - next intended action
  - bounded mainline-vs-detour interpretation where relevant
- [ ] `pass` — selected-versus-latest explanation remains available and explicit
- [ ] `pass` — prior-mainline recovery remains bounded, explainable, and testable
- [ ] `pass` — `.rules` guidance for resume-vs-start behavior is aligned with the
      implemented current contract
- [ ] `pass` — the bounded implementation does not require local auxiliary notes
      as the primary system of record for resumption

### Non-acceptance examples

- [ ] `fail` — plain `resume` still depends mainly on ad hoc local notes
- [ ] `fail` — the selected continuation target is not explainable
- [ ] `fail` — mainline-vs-detour recovery is too ambiguous to rely on

---

## 1.2 Resume reasoning remains canonical-first

### Goal

Resume behavior remains grounded in canonical workflow/checkpoint state rather
than becoming a second workflow-truth system.

### Acceptance checks

- [ ] `pass` — canonical workflow and checkpoint truth remain PostgreSQL-first
- [ ] `pass` — return-target / selected-target / task-thread interpretation is
      still treated as derived explanation rather than new canonical workflow truth
- [ ] `pass` — docs explicitly preserve the canonical/derived boundary

---

# 2. Bounded historical progress recall

## 2.1 Historical question class is explicitly supported

### Goal

The system can answer bounded questions such as:

- what was completed yesterday?
- what happened for this keyword?
- where did we stop?
- what remains?
- what did the user ask the agent about this topic yesterday?

### Acceptance checks

- [ ] `pass` — the repository documents the supported bounded historical-question
      class
- [ ] `pass` — the supported reading includes:
  - time anchor
  - keyword anchor
  - workflow/task anchor
  - bounded “what remains?” interpretation
- [ ] `pass` — unsupported broader question classes are explicitly bounded out of
      scope
- [ ] `pass` — historical answers are built from durable workflow/memory state
      rather than from local auxiliary notes as primary truth

---

## 2.2 Historical answer assembly is explainable

### Goal

Historical recall should not only work; it should explain how the answer was
assembled.

### Acceptance checks

- [ ] `pass` — bounded historical recall paths expose explicit answer-shaping or
      retrieval explanation details
- [ ] `pass` — keyword narrowing remains explainable
- [ ] `pass` — time-oriented interpretation remains bounded and documented
- [ ] `pass` — grouped primary memory surfaces remain usable for historical recall
      clients where appropriate

### Non-acceptance examples

- [ ] `fail` — historical answers are effectively opaque
- [ ] `fail` — keyword/time interpretation drifts without documentation
- [ ] `fail` — the repository implies broad historical QA beyond the current
      bounded scope

---

# 3. Automatic interaction-memory capture

## 3.1 User requests and agent responses are captured automatically

### Goal

User requests and agent responses are not only ephemeral conversation state.
They become durable interaction memory.

### Acceptance checks

- [ ] `pass` — user requests are automatically recorded as durable interaction
      records
- [ ] `pass` — agent responses are automatically recorded as durable interaction
      records
- [ ] `pass` — interaction-memory capture does not depend only on discretionary
      manual memory calls
- [ ] `pass` — interaction memory is distinguishable from other memory classes by
      type or metadata
- [ ] `pass` — the contract for what is captured automatically is documented

### Non-acceptance examples

- [ ] `fail` — interaction memory still depends mainly on ad hoc manual capture
- [ ] `fail` — user requests and agent responses are not durable enough to be
      searched later
- [ ] `fail` — interaction memory is captured but not distinguishable

---

## 3.2 Interaction memory is retrieval-ready

### Goal

Interaction memory is not only stored; it is usable for later recall.

### Acceptance checks

- [ ] `pass` — interaction memory can participate in vector-oriented search
- [ ] `pass` — interaction memory can participate in bounded graph-linking or
      relation-writing where useful
- [ ] `pass` — interaction memory can contribute to:
  - resumability
  - historical recall
  - failure-pattern avoidance
- [ ] `pass` — docs clearly describe interaction memory as:
  - durable
  - searchable
  - bounded in scope
  - not a competing canonical workflow-truth layer

---

# 4. Failure-pattern reuse and avoidance

## 4.1 Prior failure knowledge is easier to surface

### Goal

Agents should be less likely to repeat known bad patterns because prior failures
and recoveries are easier to retrieve.

### Acceptance checks

- [ ] `pass` — structured failure records remain queryable and reusable
- [ ] `pass` — prior failure patterns can be surfaced in bounded representative
      cases
- [ ] `pass` — prior recovery or workaround knowledge can be surfaced in bounded
      representative cases
- [ ] `pass` — the repository documents how prior failure knowledge should affect
      later agent decisions
- [ ] `pass` — `.rules` guidance explicitly encourages reuse of prior failure or
      recovery knowledge where appropriate

### Non-acceptance examples

- [ ] `fail` — failures are recorded but not practically reusable
- [ ] `fail` — agents still have no bounded path to surface prior similar failures
- [ ] `fail` — failure-avoidance remains an aspiration rather than an operational
      behavior surface

---

## 4.2 Repeated known bad patterns are more detectable

### Goal

The repository should make it easier to distinguish:

- fresh failure
- repeated known failure
- prior-known recovery path

### Acceptance checks

- [ ] `pass` — bounded retrieval or explanation surfaces can distinguish fresh
      versus repeated known failure
- [ ] `pass` — docs explain the intended repeated-failure reading clearly enough
      for operator and agent use
- [ ] `pass` — tests cover representative repeated-failure retrieval or explanation
      scenarios

---

# 5. File-work metadata capture

## 5.1 File-work metadata is durable and queryable

### Goal

The system should remember file-work metadata without needing to index the
contents of Git-managed files.

### Acceptance checks

- [ ] `pass` — durable memory capture includes file-work metadata such as:
  - file names
  - file paths
  - purpose of creating a file
  - purpose of modifying a file
- [ ] `pass` — file-work metadata is retrievable in bounded representative cases
- [ ] `pass` — file-work metadata can be linked, where useful, to:
  - workflows
  - checkpoints
  - interaction memory
  - failure or recovery memory
- [ ] `pass` — docs explicitly say that Git-managed file contents themselves do
      not need to become memory-search targets for `0.9.0`

### Non-acceptance examples

- [ ] `fail` — file-work metadata is not captured durably
- [ ] `fail` — the repository silently broadens scope into full file-content
      indexing without documenting the change
- [ ] `fail` — file-work purpose remains only implicit and non-queryable

# 6. `.rules` and implementation alignment

## 6.1 `.rules` reflects the actual resume and recall model

### Goal

`.rules` should act as part of the operational layer, not as drifting advisory
text.

### Acceptance checks

- [ ] `pass` — `.rules` guidance for resume behavior matches the implemented
      current canonical/derived model
- [ ] `pass` — `.rules` guidance for checkpoint discipline matches the data the
      system actually needs for later resume and recall
- [ ] `pass` — `.rules` guidance for bounded historical lookup is explicit enough
      to shape agent behavior
- [ ] `pass` — `.rules` guidance for failure avoidance is explicit enough to shape
      agent behavior
- [ ] `pass` — `.rules` guidance for interaction-memory and file-work metadata
      capture is explicit enough to matter operationally

### Non-acceptance examples

- [ ] `fail` — `.rules` and actual implementation materially disagree
- [ ] `fail` — critical resume or failure-avoidance behavior remains too ambiguous
- [ ] `fail` — `.rules` is too weak to shape agent behavior in the intended slice

# 7. Bounded operator-facing observability

## 7.1 Operator metrics remain useful and clearly defined

### Goal

Operators should be able to see whether the system is accumulating durable
memory and what posture the derived graph layer is in.

### Acceptance checks

- [ ] `pass` — operator-facing metrics remain defined and documented for:
  - canonical summary volume
  - derived graph posture
- [ ] `pass` — the current bounded metrics remain readable in the intended order:
  1. canonical relational summary volume
  2. derived graph posture
- [ ] `pass` — stats and memory-stats surfaces remain aligned with docs
- [ ] `pass` — readiness-oriented guidance remains bounded and operator-actionable

---

## 7.2 Observability remains canonical-first

### Goal

Derived graph metrics and readiness state should not be mistaken for canonical
summary truth.

### Acceptance checks

- [ ] `pass` — docs preserve the canonical-first / derived-second reading
- [ ] `pass` — a degraded, stale, or unknown graph posture is not described as
      canonical summary loss
- [ ] `pass` — observability docs remain consistent with runtime and CLI surfaces

`0.9.0` should be accepted only if the repository can honestly claim all of the
following.

- [ ] `pass` — a user can say `resume` and the agent can more reliably recover
      the intended continuation target
- [ ] `pass` — bounded historical progress questions can be answered from durable
      canonical and derived state
- [ ] `pass` — user requests and agent responses are automatically remembered in
      a retrieval-ready form

      content indexing

      reuse
- [ ] `pass` — `.rules` and `ctxledger` function together as one stronger
      operational model
- [ ] `pass` — the bounded milestone is supported by focused validation and
      aligned docs
