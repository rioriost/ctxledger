## 1. Purpose

The goal of `0.9.0` is to strengthen `ctxledger` and repository `.rules`
together so that AI-agent work becomes more reliably:

- resumable from minimal user prompts
- queryable across recent historical work
- resistant to repeating known failure patterns

This milestone is not about a broad graph-memory redesign.

It is about making the current durable workflow and memory substrate actually
behave like a practical operating system for AI-assisted software work.

In concrete product terms, `0.9.0` should move the repository closer to
supporting outcomes like:

- a user says `resume`
  - and the agent can recover the right continuation target with minimal
    ambiguity
- a user asks a human-memory-like progress question such as:
  - `"昨日"の"some keywords"はどこまで完了してる？`
  - and the system can answer from durable canonical and derived state
- historical recall is no longer treated as thin keyword filtering alone, but as
  bounded episodic recall that can combine:
  - time anchors
  - keyword/topic anchors
  - graph-linked episode relationships
  - weighting over recency and development impact
- an agent is less likely to repeat a previously observed bad pattern because
  prior failure and recovery signals are easier to surface and follow
- user requests and agent responses are automatically remembered as durable
  interaction memory that can be searched later through vector- and
  graph-supported retrieval paths
- file-content indexing remains out of scope, while metadata about file-touching
  work such as:
  - file names
  - touched paths
  - and the purpose of creating or modifying a file
  becomes durable searchable memory
- the repository can add new MCP tools beyond the current set when a new tool is
  the clearest bounded way to support resumability, historical recall, or
  failure-pattern avoidance

## 2. Why `0.9.0` should change direction

Earlier planning left `0.9.0` as the earliest milestone for broader
Mnemis-style graph-memory architectural evaluation.

That is no longer the best next step.

After `0.8.0`, the repository now already has meaningful foundations for:

- canonical workflow lifecycle state
- durable checkpoints
- episodic memory
- memory items
- embeddings
- summaries and summary memberships
- bounded task-recall
- bounded semantic search
- remember-path strengthening
- clearer primary-vs-compatibility memory context shaping
- clearer AGE readiness posture and operator metrics

Those are substantial foundations.

However, they do not yet fully close the actual user-facing product gap.

The gap is not primarily:

- “should we do a bigger graph-memory redesign now?”

The gap is:

- “does the current system reliably help the user and agent recover, inspect,
  and avoid mistakes in normal development work?”

That is why `0.9.0` should become a **ctxledger-and-`.rules` strengthening
milestone** rather than a graph-architecture exploration milestone.

## 3. Problem statement

The repository now has enough durable state to support much stronger behavior
than it originally had.

But the practical end-user experience is still weaker than the intended product
direction.

### 3.1 Resumption is still too dependent on prompt quality

The current system can store:

- workflow instances
- checkpoints
- next intended actions
- memory episodes
- memory items
- task-recall details

But a user saying only `resume` can still leave too much ambiguity around:

- which workflow should be foregrounded
- whether the latest work was a detour
- whether the selected target should be the latest work or a prior mainline
- which checkpoint should anchor the resumed task thread
- which next action should be surfaced first

### 3.2 Historical recall is still stronger for internal structure than for user-facing questions

The repository now stores enough information to answer bounded questions about
past work.

But the current system is still better at:

- storing and structuring state

than at:

- answering user-facing historical progress questions in a stable and explicit
  way

Examples of the intended question class:

- what was finished yesterday?
- what happened for this ticket?
- what is the state of work matching these keywords?
- what remains unfinished after the last checkpoint?
- which workflow was the mainline before the latest detour?
- what did the user ask the agent to do yesterday about a topic or keyword?
- what was the agent’s last answer about a task, failure, or file-touching goal?
- which past episode is most relevant to this keyword, even if it is not the
  newest one?
- which older episode still matters because its development impact is higher than
  more recent but less important work?
- which prior episode should be recalled when the user uses a variant spelling,
  paraphrase, or semantically close phrase instead of the exact stored wording?

### 3.3 Failure knowledge is still too weakly operationalized

The repository can store failures and memory.

But agents can still repeat bad patterns because failure knowledge is not yet
strongly integrated into:

- ordinary resumption behavior
- question answering
- memory retrieval
- repository `.rules`
- checkpoint and completion discipline

### 3.4 `.rules` is still too advisory in practice

The repository `.rules` already provides meaningful operational guidance.

But in the current practical system, `.rules` can still behave too much like:

- high-quality advice

instead of:

- a tightly coupled behavior-shaping layer over canonical workflow and memory
  state

For the intended product direction, `.rules` should be treated as part of the
system design, not just documentation.

## 4. Product goals

`0.9.0` should strengthen five product capabilities.

## 4.1 Resumability from minimal prompts

The system should make it more realistic that a user can say:

- `resume`
- `continue`
- `作業を再開`

and the agent can recover the right continuation target from durable state.

This does not require magical inference.
It requires stronger canonical and derived continuation signals plus stronger
behavioral guidance.

## 4.2 Durable bounded historical recall

The system should make it easier to answer bounded questions such as:

- what was completed yesterday?
- what happened around this keyword?
- what was the state of task `X`?
- where did we stop?
- what remains?

The intended recall model should be closer to how human episodic memory works:

- time anchors matter
- keyword and topic anchors matter
- older episodes can still outrank newer ones when their development impact is
  higher
- exact string match is insufficient when paraphrase or variant wording still
  points to the same episode

The goal is not broad natural-language analytics.
The goal is bounded, operationally useful historical recall grounded in durable
workflow and memory state, with graph-backed episodic keyword recall and
RAG-assisted normalization where that materially improves bounded recall quality.

## 4.3 Failure-pattern avoidance

The system should make it easier for agents to avoid repeating bad patterns by
surfacing:

- prior failures
- prior skips
- prior recovery steps
- prior operator decisions
- prior root-cause and workaround knowledge

This should not become a vague aspiration.
It should be reflected in concrete retrieval, response, and `.rules` behavior.

## 4.4 Automatic interaction memory

The system should automatically record user-agent interaction history as durable
memory rather than depending only on explicit manual memory tool use.

At the current intended `0.9.0` boundary, this means:

- user requests should become durable interaction records
- agent responses should become durable interaction records
- those interaction records should be available for:
  - keyword-oriented lookup
  - bounded historical recall
  - failure-pattern avoidance
  - resumability support
- interaction records should be representable in both:
  - vector-searchable form
  - graph-linkable form where bounded relation writing is useful
- interaction records should also be eligible for bounded keyword-node linkage so
  that later recall can traverse:
  - keyword/topic anchor
  - interaction or episode
  - related workflow/checkpoint/failure/file-work context

## 4.5 File-work metadata memory

The system should make file-touching work more queryable without indexing the
contents of Git-managed files themselves.

At the intended `0.9.0` boundary:

- Git-managed file contents do not need to become memory-search inputs
- but metadata about file-touching work should be remembered, including:
  - file names
  - file paths
  - the purpose of creating a file
  - the purpose of modifying a file
  - bounded links between that file-work metadata and:
    - workflows
    - checkpoints
    - interaction memory
    - failure/recovery memory where relevant

## 5. Non-goals

`0.9.0` is not intended to be:

- a broad Mnemis-style architectural redesign
- a graph-native replacement for PostgreSQL canonical truth
- a full general-purpose history-question-answering system
- a generic analytics layer over all repository data
- a full autonomous policy engine that makes repository `.rules` executable in a
  separate runtime
- a broad retrieval-surface explosion across every tool and resource

The milestone should remain tightly scoped to:

- resumability
- bounded historical progress recall
- failure-pattern avoidance
- automatic interaction memory
- file-work metadata capture
- stronger coupling between `ctxledger` state and repository `.rules`

---

## 6. Desired end state

At the end of `0.9.0`, the repository should be able to support the following
reading:

1. a user can give a minimal “resume” style prompt and the agent can more
   reliably recover:
   - the intended workflow
   - the likely mainline continuation target
   - the latest relevant checkpoint
   - the next intended action
   - bounded detour-vs-mainline context where relevant

2. the system can answer bounded historical progress questions from durable
   state more directly and more explicitly

3. prior failure and recovery information is easier to retrieve and more likely
   to shape agent decisions

4. user requests and agent responses are automatically captured as durable
   interaction memory and are available for later retrieval

5. file-work metadata such as touched file names and file-edit purpose is
   durably remembered without treating Git-managed file contents themselves as
   memory-search targets

6. `.rules` and `ctxledger` now work together as one integrated operational
   model for:
   - resume behavior
   - checkpoint discipline
   - continuation selection
   - historical recall
   - failure avoidance
   - interaction-memory discipline

  - what was done
  - what remains

## 8. Conceptual model for `0.9.0`

The core idea of `0.9.0` is that resumability, historical recall, and
failure-avoidance are not separate features.

They are three views over the same durable operating state.

### 8.1 Resume model

A useful resume model should combine:

- canonical workflow state
- canonical checkpoint state
- canonical verification state
- bounded derived task-recall selection
- bounded prior-mainline and detour signals
- repository `.rules` guidance about when to resume versus start new work

### 8.2 Historical recall model

A useful historical recall model should combine:

- workflow time anchors
- checkpoint summaries
- episode summaries
- memory-item content
- interaction-memory content for user requests and agent responses
- file-work metadata such as touched file paths and edit purpose
- summary-first retrieval
- keyword and scope narrowing
- bounded time-oriented interpretation
- graph-backed keyword-to-episode linkage
- weighting that can balance:
  - recency
  - development impact
  - semantic proximity
  - task relevance

This model should deliberately reflect a human-memory-like reading:
the newest episode is not always the most valuable one.
An older episode can still be the right answer if its development impact or
structural relevance is stronger.

### 8.3 Failure-avoidance model

A useful failure-avoidance model should combine:

- structured failures
- memory episodes and memory items that capture failure/recovery lessons
- bounded retrieval of similar prior mistakes
- repository `.rules` guidance that tells agents when to stop, recover, or ask
  for clarification

## 9. Primary objectives

## 9.1 Objective A — Minimal-prompt resume should become more reliable

When the user gives a minimal resume command, the system should be better at
recovering the intended continuation target.

This likely requires strengthening:

- return-target and primary-objective surfaces
- mainline-versus-detour interpretation
- selected-versus-latest explanation surfaces
- checkpoint summary and next-action extraction
- `.rules` guidance for ambiguous resume situations

## 9.2 Objective B — Historical progress lookup should become a first-class bounded path

The system should have a clearer bounded path for answering questions about what
was done and what remains.

This likely requires:

- a clearer notion of time-oriented lookup boundaries
- better workflow / episode / memory lookup composition
- explicit response details showing why a historical answer was assembled
- bounded keyword-oriented lookup that can still explain its retrieval path
- graph-backed keyword/episode linkage so recall is not limited to plain
  full-text-style filtering
- weighting over recency and development impact so older but more important
  episodes can outrank newer low-impact episodes
- RAG-assisted normalization so variant spellings, paraphrases, and semantically
  close phrases can still recover the intended prior episode

## 9.3 Objective C — Failure knowledge should become more reusable

The system should strengthen the path from:

- failure
- to memory
- to retrieval
- to future avoidance

This likely requires:

- more explicit capture of failure patterns and recovery steps
- better retrieval of similar prior failures
- stronger agent-facing guidance for using failure knowledge before repeating
  work

## 9.4 Objective D — Automatic interaction memory should become first-class

The system should strengthen the path from live user-agent interaction to durable
retrieval-ready memory.

This likely requires:

- automatic capture of user requests and agent responses
- interaction memory types or categories that remain distinguishable from other
  memory kinds
- vector-searchable interaction records
- bounded graph-linkable interaction records where relation writing is useful
- keyword/topic linkage for interaction memory where that improves later episodic
  recall
- clear rules for when interaction memory should link to:
  - workflows
  - checkpoints
  - failure records
  - file-work metadata

## 9.5 Objective E — File-work metadata capture should become durable and queryable

The system should strengthen the path from file-touching work to durable
metadata-oriented memory.

This likely requires:

- explicit capture of touched file names and paths
- explicit capture of the purpose of creating or modifying a file
- bounded links between file-work metadata and:
  - workflows
  - checkpoints
  - interaction memory
  - remembered failures or recovery patterns where relevant

## 9.6 Objective F — `.rules` should be strengthened as an operational layer

`.rules` should not remain merely descriptive.
It should become more explicitly aligned with the actual failure and recovery
modes observed in the repository.

This likely includes:

- stronger resume-versus-start guidance
- stronger checkpoint content expectations
- stronger requirements around recording what changed, what remains, and what to
  do next
- stronger guidance for reusing prior failure knowledge
- stronger guidance for bounded historical lookup requests

## 10. Proposed implementation direction

## 10.1 Resume-strengthening direction

Strengthen the current resume path so it can better recover the right target
from minimal user prompts.

Candidate directions:

- improve current mainline-versus-detour recovery
- make selected continuation target more directly inspectable from canonical and
  derived state
- expose clearer “why this workflow was selected” details
- make resume-related operator/debug payloads reflect the exact recovery basis
- align `.rules` with the actual continuation-selection behavior

## 10.2 Historical-lookup direction

Add or strengthen bounded historical lookup behavior for questions that combine:

- time anchor
- keyword anchor
- task/workflow anchor

Candidate directions:

- strengthen current retrieval contracts around:
  - recent workflows
  - checkpoint summaries
  - episode summaries
  - summary-first memory context
- add graph-backed keyword/topic nodes or equivalent bounded graph structures so
  episodic recall can traverse from keywords to episodes rather than depending
  only on plain text matching
- add weighting rules that balance:
  - recency
  - development impact
  - current task relevance
  - semantic proximity
- use RAG-assisted normalization to absorb spelling variation, paraphrase, and
  semantically close wording before bounded retrieval assembly
- add explicit explanation fields for historical answer assembly
- preserve primary grouped retrieval surfaces where possible

## 10.3 Failure-pattern direction

Strengthen the repository so prior failures become more reusable.

Candidate directions:

- improve the path from structured failure records into reusable memory
- better expose “similar prior failure” and “prior recovery pattern” retrieval
- make operator and agent surfaces distinguish:
  - fresh failure
  - repeated known failure
  - prior-known recovery path
- strengthen `.rules` guidance to prefer reuse of failure-recovery knowledge
  where applicable

## 10.4 Interaction-memory direction

Treat user-agent interaction as first-class durable memory input.

Candidate directions:

- automatically record:
  - user requests
  - agent responses
- persist those records in a form that can support:
  - vector search
  - bounded graph relations
  - keyword/topic linkage for episodic recall
- introduce bounded relations between interaction memory and:
  - workflows
  - checkpoints
  - memory items
  - failures
  - file-work metadata
- define when interaction memory should be summarized versus kept at the message
  level
- define when interaction memory should be promoted into reusable memory items or
  summary artifacts
- define when interaction memory should contribute keyword/topic anchors for
  later graph-backed episodic recall

## 10.5 File-work metadata direction

Treat file-touching work metadata as durable searchable memory even when file
contents themselves are not indexed.

Candidate directions:

- capture touched file paths and file names from agent work events
- capture file-edit purpose from:
  - checkpoint summaries
  - completion summaries
  - interaction-memory summaries
  - explicit structured metadata where available
- avoid treating Git-managed file contents as memory-search targets
- strengthen links from file-work metadata to:
  - workflows
  - checkpoints
  - interaction memory
  - failure/recovery memory

## 10.6 Tool-surface direction

The current MCP tool surface does not need to remain fixed if a bounded tool
addition is the clearest way to support the `0.9.0` goals.

Candidate directions:

- add new MCP tools when needed for:
  - automatic interaction-memory ingestion
  - bounded historical progress lookup
  - prior-failure lookup
  - resumability support
- keep additions narrow and directly tied to the milestone goals
- avoid broad tool-surface expansion unrelated to resumability, historical
  recall, or failure avoidance

## 10.7 `.rules` strengthening direction

Treat repository `.rules` as part of the product surface.

Candidate directions:

- make resume rules more explicit for:
  - plain `resume`
  - plain `continue`
  - ambiguous historical lookup requests
- make checkpoint rules more explicit for:
  - what changed
  - what remains
  - next intended action
  - verification status
  - blockers and risks
- add stronger guidance for:
  - reading canonical context first
  - bounded historical question interpretation
  - surfacing prior failure knowledge before retrying similar work
  - recording interaction memory and file-work purpose strongly enough for later
    recall

evidence for all of the following.

- a minimal “resume” style request can recover the correct or intended
  continuation target in bounded representative cases

- bounded historical progress queries can be answered from durable state

- prior failure patterns and recovery knowledge can be surfaced in bounded
  representative cases

- docs and tests show how the system is intended to reduce repeated failures

### 12.4 Interaction-memory validation

- user requests and agent responses are automatically captured in bounded
  representative cases
- those interaction records are retrievable through the intended search/recall
  paths
- interaction memory can contribute to resumability or historical recall without
  becoming a competing canonical workflow-truth layer

### 12.5 File-work metadata validation

- file names or paths touched during work can be surfaced in bounded
  representative cases
- the purpose of creating or modifying a file can be surfaced in bounded
  representative cases

- any new MCP tool added for `0.9.0` is clearly justified by a bounded product

- new tool behavior is documented and validated
- the milestone remains understandable even if the MCP tool count grows beyond
  the current set

### 12.7 `.rules` alignment validation

- `.rules` reflects the current canonical resume and recall model
- `.rules` no longer leaves important resume or failure-avoidance behavior too
  ambiguous
- `.rules` makes interaction-memory and file-work metadata expectations explicit
  enough to matter operationally
- repository guidance and actual implementation no longer drift materially in
  the bounded `0.9.0` scope

### 12.8 Full validation

- focused tests for the new slice are green
- full repository validation remains green
- operator-facing docs match the implemented current boundary

`0.9.0` is successful if the repository can honestly claim:

- a user can say `resume` and the agent can more reliably recover the right
  continuation target from durable state
- a user can ask bounded historical progress questions and get useful answers
  grounded in canonical workflow and memory state
- historical episodic recall can use graph-backed keyword/topic linkage rather
  than depending only on thin full-text-style matching
- historical episodic recall can weight recency against development impact so
  older high-value episodes can still be surfaced

- user requests and agent responses are automatically remembered in a form that
  can participate in later search and recall
- file-touching work metadata such as file names and edit purpose is durably
  queryable without requiring indexing of Git-managed file contents themselves
- agents are less likely to repeat known failure patterns because prior failure
  and recovery information is easier to retrieve and follow
- `.rules` and `ctxledger` behave like one integrated resumability and
  durable-memory operating model
- the repository is stronger at practical AI-assisted development continuity,
  not just at durable storage

## 14. Relationship to later work

After `0.9.0`, the repository will be in a better position to evaluate larger
future architecture questions if they are still warranted.

But the milestone order should remain:

- strengthen practical resumability, recall, and failure avoidance first
- evaluate larger architectural shifts later only if the product layer still
  justifies them

That means future graph-memory or retrieval-architecture exploration should be
treated as a later explicit choice, not as the default next step after `0.8.0`.

## 15. Summary

`0.9.0` should no longer be a broad Mnemis-style architectural evaluation
milestone.

It should be the milestone where `ctxledger` and repository `.rules` are
strengthened together so the system can more credibly support:

- minimal-prompt resume
- bounded historical progress recall
- failure-pattern avoidance
- automatic durable interaction memory
- durable searchable file-work metadata
- bounded MCP tool expansion where it materially improves those outcomes

This is the more practical next step for the product.

It builds directly on the durable workflow, task-recall, remember-path, and
operator-visibility foundations already in place.

And it moves the repository closer to the user-facing goal that actually
matters:

- work should be interruptible
- work should be recoverable
- work should be inspectable later
- and agents should get better over time instead of repeating the same failures
