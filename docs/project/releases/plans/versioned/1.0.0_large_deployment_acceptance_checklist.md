# `1.0.0` Large Deployment Acceptance Checklist

## Purpose

This checklist defines the release-facing acceptance criteria for the `1.0.0`
**large deployment pattern** for `ctxledger`.

The large deployment pattern is intended to add a credible Azure-hosted
deployment path that complements the existing local `small` pattern.

The intended target shape is centered on:

- Azure Container Registry
- Azure Container Apps
- Azure Database for PostgreSQL Flexible Server
- Azure OpenAI
- PostgreSQL `azure_ai` extension usage aligned to Azure OpenAI

This checklist should be used to answer the practical release question:

> Is the `1.0.0` large Azure deployment pattern complete enough to accept as a
> bounded release slice?

A key acceptance goal for this release is that the Azure `large` pattern should
provide a **self-contained one-command deployment experience**:

- `azd up` should provision infrastructure and deploy the application
- the container image used by the large pattern should be built through the
  standard `azd` remoteBuild flow against Azure Container Registry rather than
  by relying on a local Docker engine
- required PostgreSQL allowlisting, `azure_ai` bootstrap, schema bootstrap, and
  postdeploy smoke validation should complete automatically as part of that flow
- the postdeploy smoke validation should prefer an MCP `initialize` probe, and
  ideally follow it with:
  - `tools/list`
  - `resources/list`
  rather than relying only on a generic HTTP reachability check
- the deployment should create the supporting execution prerequisites it needs,
  rather than depending on precreated operator-managed bootstrap resources
- on first run, the only intended interactive choices should be:
  - Azure subscription selection
  - Azure location selection
- the terminal output at the end of `azd up` should be handoff-first, so the
  next user action is immediately obvious
- that terminal output should be README-first rather than raw-snippet-first,
  surfacing:
  - the MCP endpoint
  - the generated snippet README path
  - the snippet directory
  - the summary metadata file
- after deployment, the remaining user-facing to-do should ideally be limited to:
  - configuring the AI agent or MCP client to use the deployed endpoint
- the deployment flow should surface MCP client configuration snippets automatically
  for representative clients so the final setup step is as small as practical
- the deployment flow should also write MCP client snippet artifacts so users can
  copy from generated files instead of reconstructing the first configuration by
  hand
- the deployment flow should support auth-aware MCP snippet generation, so the
  generated snippets can reflect:
  - no auth headers
  - standard bearer `Authorization` headers
  - custom header names for future large-pattern gateway alignment
- the deployment flow should also write a small MCP snippet README artifact that
  explains the generated files and how to use them
- the deployment flow should also write richer snippet summary metadata so users
  and operators can recover the deployment handoff context later, including:
  - endpoint URL
  - server name
  - auth mode
  - auth header name where relevant
  - Azure environment name
  - Azure location
  - Azure environment type
  - postdeploy smoke status
  - postdeploy smoke probe mode
  - postdeploy smoke protocol version
  - postdeploy smoke follow-up probes
  - postdeploy smoke body preview
  - deployment timestamp
  - application version
  - generated file paths
  - a preferred artifact hint pointing to the rendered README
  - auth-aware usage hints
  - environment-aware usage hints
- the accepted large pattern should also define environment-specific auth handoff
  profiles so generated MCP client snippets can align with the intended auth
  posture per environment, and those profiles should be carried explicitly in the
  checked-in parameter files, such as:
  - `dev`
    - `mcpAuthMode=none`
    - `mcpAuthHeaderName=Authorization`
  - `staging`
    - `mcpAuthMode=bearer_header`
    - `mcpAuthHeaderName=Authorization`
  - `prod`
    - `mcpAuthMode=custom_header`
    - `mcpAuthHeaderName=X-Forwarded-Access-Token`

A critical requirement of this large pattern is that it should **not** simply
reuse the `small` pattern’s direct `OPENAI_API_KEY` to public OpenAI posture.
Instead, the accepted Azure design should maximize Azure-native AI integration:

- `ctxledger` large deployments should target **Azure OpenAI**
- Azure Database for PostgreSQL Flexible Server should use the
  **`azure_ai` extension** where appropriate
- direct embedding generation from PostgreSQL, such as
  `azure_openai.create_embeddings(...)`, should be treated as a first-class
  design capability for the large pattern

This document is intentionally strict about separating:

- **cloud deployment capability**
- **multi-user access posture**
- **true in-application authorization maturity**

The large pattern may be accepted as a deployment feature without claiming that
`ctxledger` has become a complete multi-tenant platform.

---

## Reading rules

When using this checklist:

- treat PostgreSQL as the canonical system of record
- treat `pgvector` as part of the required active retrieval path
- treat Apache AGE, when enabled, as a bounded and degradable support layer
- treat Azure OpenAI as the default large-pattern AI provider
- treat PostgreSQL `azure_ai` integration as a first-class large-pattern design
  concern, not an optional afterthought
- treat authentication and public ingress as external to `ctxledger`
- do not assume that “Azure-hosted” automatically means:
  - tenant isolation
  - per-user authorization maturity
  - enterprise-complete security architecture
- do not accept the milestone based only on documentation if the release claims
  runnable Azure support
- do not over-claim Traefik continuity from the `small` pattern into Azure
- do not accept a large deployment design that still depends primarily on direct
  public OpenAI API usage in the same way as the `small` pattern

---

## Acceptance result values

Each checklist item should be read using one of:

- `pass`
- `partial`
- `fail`
- `deferred`

The milestone should be treated as **accepted** only when:

- all required core items are `pass`
- any `partial` items are explicitly bounded and non-blocking
- deferred items are explicitly outside the bounded `1.0.0` scope
- the repository documentation does not imply broader guarantees than the
  implementation actually provides

---

## Core acceptance areas

`1.0.0` large deployment is accepted only if all of these areas are satisfied
strongly enough:

1. Azure target topology is explicit and coherent
2. container delivery path is real and repeatable
3. Azure PostgreSQL posture is validated
4. `pgvector`, `azure_ai`, Azure OpenAI, and Apache AGE expectations are handled correctly
5. ingress and authentication boundary are explicit
6. runtime scaling and database connection posture are bounded
7. secret management and cloud security posture are acceptable
8. observability and operational diagnosis are adequate
9. deployment and rollback procedures are defined
10. documentation and release-readiness are aligned
11. one-command `azd up` experience is materially achieved
12. staging and production parameter profiles are present and aligned with the accepted large-pattern posture

- Azure location selection
