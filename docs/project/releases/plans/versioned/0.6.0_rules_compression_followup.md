# `0.6.0` Rules Compression and Command-Style Follow-Up

## Purpose

This note records the bounded follow-up work needed after the initial `.rules`
update for the `0.6.0` hierarchical memory slice.

It addresses three practical concerns:

1. removing unnecessary version-specific wording from `.rules`
2. making the rules easier for AI agents to follow consistently
3. measuring and reducing rule length in the most agent-relevant section

This is **not** a new feature plan for hierarchical memory itself.

It is a focused repository-guidance follow-up for the shape and usability of the
agent rules.

---

## Problem statement

The earlier `.rules` follow-up correctly captured the new meaning of the richer
MCP payloads, especially around:

- `memory_get_context`
- grouped hierarchy-aware output
- retrieval-route metadata
- workflow completion `auto_memory_details.summary_build`

However, that first pass left three issues:

### 1. Version wording was too explicit

The `.rules` file is intended to guide future agents operating on the **current**
repository state.

It does not need repeated in-line wording like:

- “for the current `0.6.0` slice”
- “for the current `0.6.0` narrow path”

when the repository has already accepted that slice and the rules are supposed to
describe what is true **now**.

That wording adds noise without improving behavior.

### 2. The guidance was descriptive before it was operational

The earlier wording explained the behavior well, but it was still more prose-like
than command-like.

For AI-agent-facing rules, a more durable style is:

- read X first
- treat Y as canonical
- never infer Z from A
- keep B and C separate

This is easier to obey than long explanatory prose.

### 3. The most agent-relevant section became too long

The `.rules` section beginning at:

- `# General Agent Workflow Rules for Repositories Using ctxledger`

is the main block future agents are expected to consume and obey.

If that section grows too much, then even good guidance can become less effective
because:

- the signal-to-noise ratio drops
- important distinctions get buried
- agents may follow only the obvious early bullets
- repetition competes with actual instructions

So the follow-up needs not only to improve wording, but also to tighten length.

---

## Scope of this follow-up

This follow-up is intentionally limited to:

- `.rules`
- the agent-facing interpretation of current MCP payloads
- wording structure and compression
- explicit command-style guidance for the most important distinctions

It does **not** propose changing:

- current runtime behavior
- current MCP schemas
- `memory_get_context` semantics
- summary automation semantics
- graph/retrieval boundaries

The implementation is purely a guidance and maintainability refinement.

---

## Current payload interpretation needs that must be preserved

Any compression must preserve the most important distinctions from the earlier
rules update.

## 1. `memory_get_context` is no longer a flat lookup

Agents must still understand that:

- `memory_get_context` is hierarchy-aware
- grouped output matters
- route metadata matters
- canonical summary-first selection and episode-derived fallback are distinct

## 2. `memory_context_groups` remains primary when structure matters

Compression must not regress to a reading where only flat convenience fields are
used.

## 3. `graph_summary_auxiliary` is still derived, not canonical

This is one of the highest-value safeguards and must remain explicit.

## 4. `include_episodes = false` still needs narrow reading

Agents must not infer hidden summary-first state or placeholder fields when the
response does not actually surface them.

## 5. `auto_memory_details.summary_build` still needs literal reading

Workflow completion summaries must continue to distinguish:

- auto-memory recorded
- summary build attempted
- summary build succeeded

These are separate facts and should remain separate in the rules.

---

## Compression goals

The follow-up should satisfy all of the following at once.

### Goal A — remove unnecessary version references

The rules should describe current repository behavior directly rather than
repeating explicit `0.6.0` labels throughout the operational bullets.

### Goal B — shift to command-style wording

Prefer imperative guidance such as:

- read structured fields first
- use grouped output when structure matters
- never treat graph-backed auxiliary context as canonical truth
- keep auto-memory and summary-build facts separate

### Goal C — reduce repetition

Avoid restating the same ideas in multiple verbose forms.

In particular, avoid repeating all of the following more than needed:

- “current stage”
- “current slice”
- “current path”
- “should be read as”
- “do not collapse”
- “do not treat as”

These ideas are still useful, but they should be expressed once in a stronger and
shorter form.

### Goal D — preserve the highest-signal distinctions

The compression must not remove the rules that actually prevent future mistakes.

## Measured size of the current agent-rules section

The section beginning at:

- `# General Agent Workflow Rules for Repositories Using ctxledger`

was measured in its current expanded form.

### Measured size

- approximately **11,905 characters**
- approximately **1,648 whitespace-separated words**

This is not a tokenizer-accurate model-specific token count, but it is a useful
repository-local proxy for rule length and reading burden.

### Interpretation

This size is not catastrophic, but it is large enough that compression is
worthwhile.

The problem is less “the file is too big to exist” and more:

- the densest new guidance was added in long-form prose
- the highest-value rules are harder to scan quickly
- the agent-facing section should carry a higher instruction-to-explanation
  ratio

## Recommended command-style rewrite pattern

The best rewrite pattern for `.rules` is:

### 1. state the rule as an action
Example shape:

- Treat `memory_get_context` as hierarchy-aware.
- Read structured fields before prose summaries.
- Use grouped output when structure matters.
- Never treat `graph_summary_auxiliary` as canonical truth.

### 2. follow with a short list of the exact fields or cases
Example shape:

- Read:
  - `summary_selection_applied`
  - `summary_selection_kind`
  - `memory_context_groups`
  - route metadata

### 3. avoid explanatory paragraphs unless the distinction is subtle
For subtle cases, one compact sentence is usually enough.

Example shape:

- If `include_episodes = false`, read only the surfaced fields and routes; do
  not assume hidden summary-first placeholders.

This keeps the rules operational.

## Recommended compression approach by subsection

## 1. Memory tool usage

### Keep
Keep the following concepts explicitly:

- `memory_get_context` is hierarchy-aware
- use structured fields first
- use grouped output first when structure matters
- read route metadata literally
- `graph_summary_auxiliary` is derived, not canonical
- `summary_selection_kind` distinguishes canonical summary-first vs episode
  fallback
- `include_episodes = false` is a narrow path

### Compress
Instead of long prose, rewrite into a sequence such as:

- Treat `memory_get_context` as hierarchy-aware, not flat.
- Read structured fields first:
  - ...
- Use `memory_context_groups` when context structure matters.
- Read routes literally:
  - ...
- Never treat `graph_summary_auxiliary` as canonical truth.
- If `include_episodes = false`, read only surfaced fields and routes.

This keeps almost all of the value with fewer words.

## 2. Agent behavior expectations

### Keep
Keep the following concepts explicitly:

- prefer structured fields over short status messages
- inspect `auto_memory_details`
- inspect nested `summary_build`
- distinguish:
  - auto-memory recorded
  - summary build attempted
  - summary build succeeded
- current summary automation is explicit, gated, and non-fatal
- policy metadata should be read literally

### Compress
Rewrite into a short command-style cluster such as:

- Prefer structured MCP/tool fields over prose summaries.
- If `auto_memory_details` is present, inspect it before summarizing.
- If `summary_build` is present, read it literally:
  - ...
- Keep these facts separate:
  - ...
- Read policy metadata literally:
  - ...

This is both shorter and easier to obey.

## Recommended compressed rules shape

A good compressed shape for the newly added guidance is roughly:

### Memory tool usage
- Treat `memory_get_context` as hierarchy-aware, not flat.
- Read structured fields first:
  - `summary_selection_applied`
  - `summary_selection_kind`
  - `memory_context_groups`
  - route metadata
  - summary-first mode fields
- Use `memory_context_groups` when structure matters.
- Read routes literally:
  - `summary_first`
  - `episode_direct`
  - `workspace_inherited_auxiliary`
  - `relation_supports_auxiliary`
  - `graph_summary_auxiliary`
- Never treat `graph_summary_auxiliary` as canonical truth.
- Read summary state as canonical-relational first; read graph-backed summary
  enrichment as derived and degradable.
- If `include_episodes = false`, read only surfaced fields and routes.
- Interpret:
  - `memory_summary_first` as canonical summary-first selection
  - `episode_summary_first` as episode-derived summary fallback

### Agent behavior expectations
- Prefer structured MCP/tool fields over brief human-readable status messages.
- If `auto_memory_details` is present, inspect it before summarizing.
- If `summary_build` is present, treat it as authoritative.
- Read at least:
  - attempted
  - succeeded
  - requested
  - status
  - skipped reason
  - trigger
  - scope
  - replacement
  - built summary id
  - membership count
- Keep these facts separate:
  - auto-memory recorded
  - summary build attempted
  - summary build succeeded
- Read policy metadata literally:
  - `default_requested = false`
  - `request_field = latest_checkpoint.checkpoint_json.build_episode_summary`
  - `trigger = latest_checkpoint.build_episode_summary_true`

This is the right balance between fidelity and compression.

## Acceptance criteria for this follow-up

This compression follow-up should be considered complete when:

1. `.rules` no longer uses unnecessary `0.6.0` version wording in the operational
   payload-interpretation bullets
2. the new guidance reads more like commands than prose explanation
3. the highest-value distinctions remain preserved:
   - hierarchy-aware `memory_get_context`
   - grouped output priority
   - route-literal reading
   - graph-derived-not-canonical reading
   - narrow `include_episodes = false` reading
   - `summary_build` literal reading
   - auto-memory vs summary-build fact separation
4. the general agent-rules section is measurably smaller in character/word terms
   than the expanded prose-heavy version
5. the guidance remains consistent with:
   - `docs/memory/design/memory_get_context_service_contract.md`
   - workflow completion summary-build payload behavior
   - the accepted bounded `0.6.0` slice

## Why this is a good bounded follow-up

This is a good bounded cleanup because it improves:

- agent obedience
- payload interpretation accuracy
- rule readability
- future maintainability

without changing:

- implementation behavior
- schemas
- contracts
- acceptance boundaries

In other words, it is a **guidance-quality improvement**, not a hidden product
change.

---

## Final recommendation

Implement the compressed command-style rewrite now.

Use the current expanded version only as source material for the command-style
rules, not as the final wording to preserve verbatim.

The strongest pattern going forward is:

- keep `.rules` operational
- keep examples and deep explanation in docs
- keep only the highest-signal distinctions in the agent-facing rule file

That pattern should make future agent behavior more reliable while keeping rule
bloat under control.
