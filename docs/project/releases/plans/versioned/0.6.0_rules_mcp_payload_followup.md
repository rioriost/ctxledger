# `0.6.0` Rules Follow-Up for MCP Payload Evolution

## Purpose

This note records the bounded `0.6.0` follow-up work needed to align
agent-facing repository rules with the current MCP payloads exposed by the
implemented hierarchical-memory slice.

It is intended to answer a narrow operational question:

> Now that `0.6.0` changed the meaning and richness of certain MCP payloads,
> what should the repository rules say so that agents interpret those payloads
> correctly instead of flattening them back into older assumptions?

This document is **not** a new feature plan for hierarchical memory itself.

It is a follow-up note for:

- agent guidance
- rule wording
- payload interpretation discipline
- safe operational use of the current `0.6.0` MCP/tool outputs

---

## Context

The bounded `0.6.0` slice changed the practical meaning of two important MCP
surfaces:

1. `memory_get_context`
2. workflow-completion payloads containing `auto_memory_details`

The tool names are mostly unchanged.
What changed is the **semantic richness** of the returned structured data.

That creates a risk:

- the implementation may now be correct
- but agents following older or overly generic repository rules may still behave
  as if:
  - `memory_get_context` were only a flat episode lookup
  - workflow completion were only a terminal state update with optional vague
    memory side effects

This follow-up exists to reduce that gap.

---

## Why this follow-up is needed

Recent `0.6.0` work introduced:

- summary-first context selection
- direct summary-member memory-item expansion
- grouped hierarchy-aware output
- retrieval-route metadata
- explicit distinction between primary and auxiliary context routes
- narrower `include_episodes = false` shaping
- richer workflow-completion automation metadata
- explicit `summary_build` result details nested under `auto_memory_details`

Without a rules update, agents are likely to keep doing things like:

- treat `memory_get_context` as a binary “context present / absent” tool
- ignore `memory_context_groups`
- ignore `summary_selection_kind`
- ignore retrieval route metadata
- misread graph-backed auxiliary context as canonical truth
- conflate:
  - auto-memory recorded
  - summary build attempted
  - summary build succeeded
- incorrectly claim that summary build happened merely because workflow auto
  memory was recorded

Those are now avoidable mistakes.

---

## Current `0.6.0` payload changes that matter

## 1. `memory_get_context` is now hierarchy-aware

The current `0.6.0` implementation no longer behaves like a purely flat
episode-oriented helper.

It now includes a constrained hierarchy-aware retrieval shape with:

- canonical summary-first selection when available
- direct summary-member memory-item expansion
- grouped output through `memory_context_groups`
- additive retrieval-route explainability
- bounded graph-backed auxiliary enrichment

### Important current fields

The current payload can include fields such as:

- `summary_selection_applied`
- `summary_selection_kind`
- `memory_context_groups`
- `retrieval_routes_present`
- `primary_retrieval_routes_present`
- `auxiliary_retrieval_routes_present`
- `retrieval_route_presence`
- `retrieval_route_group_counts`
- `retrieval_route_item_counts`
- `summary_first_has_episode_groups`
- `summary_first_is_summary_only`
- `summary_first_child_episode_count`
- `summary_first_child_episode_ids`
- `primary_episode_groups_present_after_query_filter`
- `auxiliary_only_after_query_filter`

These are not decorative metadata.
They materially explain what retrieval path was used.

---

## 2. `memory_context_groups` is now the primary hierarchy-aware surface

The current contract direction treats:

- `memory_context_groups`

as the primary grouped hierarchy-aware response surface.

This matters because an agent that still reads only flat top-level convenience
fields may miss:

- summary grouping
- child episode presence
- auxiliary route boundaries
- the difference between summary-only primary shaping and auxiliary-only output

The rules should therefore guide agents to read grouped output first when the
task depends on context structure.

---

## 3. Retrieval routes now carry real operational meaning

The current route set includes:

- `summary_first`
- `episode_direct`
- `workspace_inherited_auxiliary`
- `relation_supports_auxiliary`
- `graph_summary_auxiliary`

The rules should explicitly preserve these interpretations:

### `summary_first`
Primary compressed path through summary-first selection.

### `episode_direct`
Direct visible episode path.

### `workspace_inherited_auxiliary`
Inherited workspace-side auxiliary context.

### `relation_supports_auxiliary`
Relation-derived auxiliary context.

### `graph_summary_auxiliary`
Derived graph-backed auxiliary summary-member context.

The last one is especially important because it is easy to overread.

The current slice requires that agents **must not** treat
`graph_summary_auxiliary` as canonical truth.

---

## 4. `include_episodes = false` remains intentionally narrow

A key `0.6.0` contract choice is that the episode-less path remains narrower than
summary-only primary shaping.

That means agents should **not** assume:

- summary-first grouped output is still visible
- summary-first explanation fields appear as false placeholders
- an episode-less response can be read as if it were just a summary-only primary
  response with child groups suppressed

The rules should explicitly tell agents to read only the surfaced fields and
routes that are actually present.

---

## 5. Workflow completion payloads now expose bounded summary automation details

The workflow-completion path can now return:

- `auto_memory_details`

and, when relevant:

- `auto_memory_details.summary_build`

This nested payload contains real operational distinctions such as:

- `summary_build_attempted`
- `summary_build_succeeded`
- `summary_build_requested`
- `summary_build_status`
- `summary_build_skipped_reason`
- `summary_build_trigger`
- `summary_build_scope`
- `summary_build_kind`
- `summary_build_replace_existing`
- `summary_build_non_fatal`
- `summary_build_replaced_existing_summary`
- `built_memory_summary_id`
- `built_summary_kind`
- `built_summary_membership_count`

These distinctions matter.

An agent should no longer compress this into vague language like:

- “auto memory ran”
- “summary build happened”
- “workflow completion created a summary”

unless the structured payload actually supports that conclusion.

---

## Desired rule update outcome

The repository rules should be updated so that a future agent naturally does the
following:

### A. Read structured payloads literally
Prefer structured fields over vague inference from status messages.

### B. Treat `memory_get_context` as hierarchy-aware
Stop treating it as only a flat episode lookup.

### C. Use route metadata when retrieval source matters
Do not guess whether context came from canonical summary selection, fallback,
workspace inheritance, relation support, or graph-backed auxiliary enrichment.

### D. Treat grouped output as primary when structure matters
Especially when reasoning about summaries, child episodes, or route boundaries.

### E. Distinguish workflow auto-memory from summary-build success
These are now separate facts and should be summarized separately.

### F. Keep graph-backed enrichment subordinate to relational truth
Do not overstate graph-backed auxiliary context as canonical correctness.

---

## Proposed rule additions

The rule update should add guidance in two main areas.

## 1. Memory tool usage guidance

The rules should explicitly say that for the current `0.6.0` slice:

- `memory_get_context` is hierarchy-aware
- `memory_context_groups` is the primary grouped hierarchy-aware surface
- retrieval-route metadata should be used when context origin matters
- summary-first sub-modes should be read explicitly
- episode-less shaping should be interpreted narrowly and literally

### Recommended wording themes

- treat `memory_get_context` as a hierarchy-aware, route-explainable tool
- pay attention to `summary_selection_kind` and route metadata
- do not collapse summary-only primary shaping into auxiliary-only shaping
- do not assume summary-first grouped output remains visible when
  `include_episodes = false`
- do not treat `graph_summary_auxiliary` as canonical truth

## 2. Agent behavior expectations for workflow-completion payloads

The rules should explicitly say that if a workflow-completion result includes:

- `auto_memory_details`

the agent should inspect it before summarizing what happened.

And if it includes:

- `auto_memory_details.summary_build`

the agent should treat that nested payload as the authoritative explanation of:

- whether summary building was attempted
- whether it succeeded
- whether it was requested
- what trigger applied
- why it was skipped
- what artifact was produced

### Recommended wording themes

- prefer structured fields over human-readable summary text
- distinguish:
  - auto-memory recorded
  - summary build attempted
  - summary build succeeded
- treat current workflow-summary automation as:
  - explicit
  - checkpoint-gated
  - non-fatal
- read policy metadata literally:
  - `default_requested = false`
  - `request_field = latest_checkpoint.checkpoint_json.build_episode_summary`
  - `trigger = latest_checkpoint.build_episode_summary_true`

---

## What this follow-up should not do

This rules follow-up should **not**:

- redefine the `0.6.0` implementation
- broaden `0.6.0` scope
- introduce new MCP tools
- treat graph-backed summary enrichment as canonical
- force agents to over-explain every response unnecessarily
- replace actual payload reading with a new vague prose-only policy

The point is to improve **payload literacy**, not to create a second layer of
ambiguous abstraction.

---

## Implementation recommendation

The rule update should be implemented directly in:

- `.rules`

Recommended placement:

1. under **Memory tool usage**
   - for `memory_get_context` hierarchy-aware payload interpretation
2. under **Agent behavior expectations**
   - for workflow-completion payload interpretation and structured-field reading

This keeps the guidance close to the behavior it governs.

---

## Acceptance criteria for this follow-up

This follow-up should be considered complete when:

1. `.rules` explicitly reflects current `0.6.0` `memory_get_context` payload
   semantics
2. `.rules` explicitly reflects current workflow-completion summary-build payload
   semantics
3. the wording preserves the canonical relational-versus-derived graph boundary
4. the wording does not overclaim new functionality
5. a future agent reading `.rules` is less likely to:
   - flatten grouped context into generic text
   - misread auxiliary graph context as canonical truth
   - confuse auto-memory recording with summary-build success

---

## Expected benefit

If this follow-up is implemented well, future sessions should show fewer mistakes
such as:

- incorrect claims about what `memory_get_context` “really returned”
- missed use of grouped context
- incorrect treatment of graph-backed auxiliary context
- incorrect release summaries around workflow summary automation
- vague or inaccurate statements about summary build success

That is a worthwhile bounded `0.6.0` cleanup because it directly improves the
quality of agent behavior against the actual implemented MCP payloads.
