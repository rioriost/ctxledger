# `0.9.0` Focused Validation Plan

## Purpose

This plan defines the **focused validation frame** for the bounded `0.9.0`
milestone.

It exists to keep the milestone implementation anchored to concrete,
reviewable, test-backed behavior instead of drifting into:
- planning-only progress
- overly broad feature claims
- implicit contracts that are hard to verify later

The goal is practical:

> before `0.9.0` is treated as complete, what focused validation slices must
> exist so the repository can honestly claim the milestone behavior works?

This is not a full repository-wide QA strategy document.
It is the bounded validation plan for the `0.9.0` slice.

---

## Scope

This validation plan covers five focused areas:

1. minimal-prompt resume behavior
2. bounded historical progress recall
3. automatic interaction-memory capture
4. failure-pattern reuse and avoidance
5. file-work metadata capture and retrieval

These are the same five product-strengthening areas that define the bounded
`0.9.0` direction.

This plan also covers:
- response-shaping validation where explanation surfaces matter
- MCP / HTTP / service-layer validation where the contract crosses layers
- release-facing acceptance alignment with:
  - `0.9.0_acceptance_checklist.md`
  - `0.9.0_ctxledger_rules_strengthening_plan.md`

---

## Validation posture

`0.9.0` validation should follow these rules:

- prefer **focused scenario tests** over broad ambiguous end-to-end claims
- validate the **bounded contract** that the repository actually intends to
  support
- preserve the canonical-first reading:
  - workflow / checkpoint / failure truth is primary
  - memory, ranking, interaction memory, and graph-backed linkage remain
    derived support layers unless explicitly stated otherwise
- validate explanation surfaces when the product promise depends on
  explainability
- do not claim support for broad unconstrained historical QA
- do not broaden scope into file-content indexing just because file-work
  metadata is in scope

The desired test style for `0.9.0` is:
- narrow
- scenario-based
- contract-oriented
- explanation-aware
- durable-state-aware

---

## Why focused validation must come early

The `0.9.0` milestone is easy to weaken if validation comes too late.

Without an early focused validation frame, the repository risks:
- adding partial behavior without fixing the contract boundary
- documenting intent that is not actually enforced by tests
- changing serializer or explanation output silently
- making stronger user-facing claims than the runtime really supports

That is why the focused validation frame should be treated as part of the
implementation surface, not a post-hoc polish task.

---

## Validation layers

The milestone should be validated across three main layers.

### 1. Service-layer validation

Use service-layer tests for:
- selection logic
- retrieval shaping
- ranking and bounded narrowing behavior
- automatic memory-capture decisions
- file-work metadata assembly rules
- repeated-failure interpretation logic

This layer should carry most of the contract logic.

### 2. Transport / protocol validation

Use MCP / HTTP tests for:
- request parsing
- boundary-level capture behavior
- response serialization
- error shaping
- explanation-surface visibility
- compatibility aliases where still needed

This layer should prove that the bounded contract is visible to clients.

### 3. Repository-wide validation

Use the full test suite to confirm:
- no regression to existing workflow and memory behavior
- no breakage to established CLI / HTTP / MCP surfaces
- no accidental drift in canonical-versus-derived reading
- green validation for the bounded release slice

## Validation area 1 — Minimal-prompt resumability

## Goal

Validate that a minimal prompt such as:
- `resume`
- `continue`
- `作業を再開`

can be grounded in workspace-context durable state strongly enough to recover a
bounded continuation target.

This validation area is about the **workspace-context selection contract**.

It is not only about `workflow_resume`.
It is about the stronger question:

> from workspace context, which workflow or task thread should be resumed, and
> why?

---

## What must be validated

### Core contract points

Focused validation should prove:

- the workspace-context entry point exists as a distinct resume-selection layer
- `workflow_resume` remains the workflow-id-specific path
- selected-versus-latest reasoning is explicit
- mainline-versus-detour reasoning is explicit
- terminal latest workflows can be deprioritized in favor of non-terminal
  candidates when the bounded contract says they should
- detour-like latest workflows can be deprioritized in favor of more mainline
  candidates when the bounded contract says they should
- running workflows remain strongly preferred when appropriate
- explanation surfaces remain available and stable enough for client use

### Required explanation surfaces

Validation should cover the presence and correctness of bounded explanation
fields such as:

- selected workflow id
- latest workflow id
- running workflow id where present
- selected reason
- latest deprioritized signal
- selected-equals-latest signal
- selected-equals-running signal
- detour override signal
- ranking details
- explanation lists
- candidate comparison surfaces where applicable

### Representative scenarios

At minimum, add or keep focused tests for cases like:

1. **running workflow preferred**
   - workspace has both running and latest candidates
   - running workflow is selected
   - explanation shows why

2. **latest selected when no running candidate exists**
   - latest workflow is non-terminal and mainline-like
   - latest workflow is selected
   - resumability explanations are surfaced

3. **older non-terminal candidate selected over latest terminal**
   - latest workflow is terminal
   - older non-terminal candidate exists
   - older non-terminal candidate is selected
   - explanation shows terminal override

4. **older non-detour candidate selected over latest detour**
   - latest workflow or latest checkpoint looks detour-like
   - a stronger non-detour candidate exists
   - explanation shows detour override

5. **selected vs latest comparison remains stable**
   - comparison payload remains explicit and test-backed
   - compatibility aliases remain tested if still exposed

### Suggested validation locations

Examples of likely test homes:

- `tests/runtime/test_task_recall_workspace_resume.py`
- `tests/http/test_coverage_targets_http_workspace_resume_resource.py`
- `tests/cli/test_cli_resume.py`
- any service-layer resume-selection tests that own the current heuristic logic

### Acceptance signal

This area is strong enough when the repository can show:
- the workspace-context resume contract is explicit
- the explanation surface is stable
- the bounded heuristics are test-backed rather than implicit

## Validation area 2 — Bounded historical progress recall

## Goal

Validate that the repository supports a bounded historical-question class such
as:

- what was completed yesterday?
- what happened for this keyword?
- where did we stop?
- what remains?
- what did the user ask the agent about this topic yesterday?
- what did the agent answer about this task?

This validation area is about the **contract boundary** and its shaping logic.

It is not a claim of unconstrained historical QA.

---

## What must be validated

### Core contract points

Focused validation should prove:

- the supported historical-question class is documented and bounded
- time anchors are supported in a bounded way where implemented
- keyword or topic narrowing is explicit and explainable
- workflow / task anchor reading remains available
- “what remains?” style interpretation remains bounded
- historical assembly stays canonical-first
- derived layers enrich the answer without becoming a competing truth system

### Required shaping / explanation surfaces

Validation should cover the existence and bounded correctness of surfaces such
as:

- query normalization details
- supported anchor interpretation details
- grouped primary memory context surfaces where applicable
- selected workflow / checkpoint details where task recall contributes
- explanation of whether narrowing was query-based, workflow-based, ticket-based,
  or summary-first
- explicit absence of over-broad claims

### Representative scenarios

At minimum, add focused tests for cases like:

1. **keyword-anchored historical lookup**
   - query narrows durable state by a stored topic or phrase
   - explanation shows the bounded narrowing path

2. **workflow-anchored historical lookup**
   - retrieval scoped to a specific workflow or task
   - answer shaping remains canonical-first

3. **“where did we stop?” interpretation**
   - latest checkpoint / selected continuation context is surfaced
   - next intended action or checkpoint summary is visible where available

4. **“what remains?” interpretation**
   - bounded remainder reading is assembled from latest checkpoint / memory
   - response does not imply omniscient project planning

5. **historical lookup with grouped primary output**
   - grouped memory surfaces remain primary where the contract says they should
   - compatibility-oriented surfaces remain secondary

### Suggested validation locations

Examples of likely test homes:

- `tests/memory/test_service_context_query.py`
- `tests/memory/test_service_context_serialization.py`
- `tests/memory/test_service_context_details_task_recall_basics.py`
- `tests/memory/test_service_context_details_task_recall_ordering.py`
- HTTP / MCP tests that validate bounded response shaping for context retrieval

### Acceptance signal

This area is strong enough when the repository can show:
- a bounded historical-question class exists
- the answer-shaping contract is explicit
- the response is explainable
- the implementation does not imply broad unconstrained historical QA

## Validation area 3 — Automatic interaction-memory capture

## Goal

Validate that user requests and agent responses become durable interaction
memory through an automatic capture path rather than depending only on
discretionary manual memory calls.

For `0.9.0`, the opening design posture should be:

- start with the MCP / HTTP boundary
- define where user-side and agent-side capture happens
- keep interaction memory durable, searchable, and bounded
- preserve the canonical workflow-truth boundary

---

## What must be validated

### Core contract points

Focused validation should prove:

- there is a bounded automatic capture policy for:
  - user requests
  - agent responses
- the capture point is explicit at the boundary layer where chosen
- interaction memory is distinguishable from other memory classes by:
  - type
  - provenance
  - metadata
  - or a stable equivalent
- capture is retrieval-ready later
- interaction memory does not silently redefine canonical workflow state

### Required capture metadata themes

Validation should cover the presence or stability of metadata such as:

- interaction role:
  - user
  - agent
- content
- capture source:
  - MCP
  - HTTP
  - resource read
  - tool call
  - or equivalent bounded source signal
- workflow / workspace linkage when available
- ticket linkage when available
- request or response kind
- topic hints where applicable
- file-work intent where present

### Representative scenarios

At minimum, add focused tests for cases like:

1. **user tool call is captured automatically**
   - incoming user request produces a durable interaction-memory record
   - record preserves role and source metadata

2. **agent response is captured automatically**
   - successful response produces a durable interaction-memory record
   - record preserves role and response shape metadata

3. **capture remains bounded**
   - lifecycle noise or unsupported message classes are not over-claimed as
     interaction memory unless explicitly in scope

4. **interaction memory is distinguishable**
   - later retrieval can identify the record as interaction memory

5. **interaction memory is compatible with search / context paths**
   - interaction-memory records can participate in bounded retrieval surfaces

### Suggested validation locations

Likely validation homes depend on the implementation slice, but should include:

- MCP handler tests
- HTTP transport tests
- memory service or bridge tests for durable persistence shape
- serializer tests if interaction-memory records are surfaced directly
- future targeted integration tests if automatic capture crosses persistence and
  retrieval layers

### Acceptance signal

This area is strong enough when the repository can show:
- automatic interaction-memory capture exists
- the capture point is explicit
- the stored records are distinguishable and retrieval-ready
- the boundary does not replace canonical workflow truth

## Validation area 4 — Failure-pattern reuse and avoidance

## Goal

Validate that prior failures and recoveries are easier to surface, compare, and
reuse so agents are less likely to repeat known bad patterns.

This area is not only about storing failures.
It is about making failure knowledge operationally reusable.

---

## What must be validated

### Core contract points

Focused validation should prove:

- structured failure knowledge remains queryable
- repeated-known-failure reading is distinguishable from fresh failure reading
- prior recovery knowledge can be surfaced in representative bounded cases
- failure reuse can enrich:
  - resumability
  - historical recall
  - bounded operator reasoning
- interaction memory and file-work metadata can contribute where appropriate

### Required explanation surfaces

Validation should cover bounded output details such as:

- whether a surfaced failure looks repeated or fresh
- what prior recovery path or workaround is associated
- what source layers contributed:
  - canonical failure state
  - memory items
  - interaction memory
  - file-work metadata
- why the answer was shaped that way

### Representative scenarios

At minimum, add focused tests for cases like:

1. **known repeated failure can be surfaced**
   - prior similar failure exists
   - retrieval path identifies it as reusable prior knowledge

2. **prior recovery path can be surfaced**
   - a workaround or successful correction exists
   - response includes bounded explanation of that recovery relevance

3. **failure knowledge influences historical answer shaping**
   - a bounded historical question includes relevant prior failure context
   - explanation remains explicit

4. **failure knowledge influences resume support**
   - resume-oriented context can surface prior blocker or recovery notes
   - response does not invent a second canonical workflow truth

5. **fresh vs repeated distinction remains explainable**
   - repeated-known-failure interpretation is not opaque

### Suggested validation locations

Examples of likely validation homes:

- `docs/memory/design/failure_reuse_contract.md` aligned tests
- memory search / context tests
- any failure-lookup or failure-shaping service tests
- serializer and response tests where repeated-failure interpretation is visible

### Acceptance signal

This area is strong enough when the repository can show:
- prior failure knowledge is reusable in bounded ways
- repeated vs fresh interpretation is explicit enough to test
- failure reuse shapes later behavior rather than remaining aspirational

## Validation area 5 — File-work metadata capture and retrieval

## Goal

Validate that file-touching work becomes durably queryable through metadata
without indexing Git-managed file contents themselves.

This area is one of the easiest places for scope drift, so the validation plan
must keep the boundary explicit.

---

## What must be validated

### Core contract points

Focused validation should prove:

- file-work metadata is durably captured
- the minimum bounded schema is explicit enough to support retrieval
- file-work metadata can be linked to workflow / checkpoint context where
  available
- file-work metadata can participate in interaction-memory and failure-reuse
  reading where useful
- Git-managed file contents themselves remain out of scope for search indexing
  in `0.9.0`

### Minimum schema themes to validate

The bounded `0.9.0` validation frame should cover metadata fields equivalent to:

- `file_name`
- `file_path`
- `file_operation`
  - `create`
  - `modify`
- `purpose`
- optional `workspace_id`
- optional `workflow_instance_id`
- optional `attempt_id`
- optional `checkpoint_id`
- optional interaction or failure linkage metadata

The exact field names may evolve, but the durable reading should remain stable.

### Representative scenarios

At minimum, add focused tests for cases like:

1. **file-work metadata captured for a create action**
   - file name, path, operation, and purpose are preserved

2. **file-work metadata captured for a modify action**
   - modify intent is preserved distinctly from create intent

3. **file-work metadata linked to workflow or checkpoint**
   - later retrieval can associate file-work activity with task progress

4. **file-work metadata appears in bounded historical or failure retrieval**
   - retrieval can surface why a file was touched without indexing file contents

5. **scope boundary remains explicit**
   - tests do not imply full file-content indexing

### Suggested validation locations

Likely validation homes include:

- memory service tests for durable metadata creation
- search / context shaping tests for retrieval visibility
- MCP / HTTP tests if boundary-level capture populates file-work metadata
- future integration tests once the persistence slice is implemented

### Acceptance signal

This area is strong enough when the repository can show:
- file-work metadata is durable
- retrieval can use it in bounded representative cases
- the implementation does not drift into file-content indexing

## Cross-cutting validation requirements

The five focused areas above are the main slices, but `0.9.0` also requires
cross-cutting validation.

## 1. Serializer and response-shaping coverage

Add or update focused tests whenever the milestone changes:
- response fields
- explanation details
- grouped primary surfaces
- compatibility aliases
- capture metadata visibility
- repeated-failure or file-work metadata shaping

If a user-facing contract is meaningful, it should not be left untested at the
serialization boundary.

## 2. MCP / HTTP boundary coverage

Add focused tests when `0.9.0` behavior is decided or enforced at:
- MCP tool call handling
- MCP resource reads
- HTTP endpoint handling
- streamable HTTP request / response shaping

This matters especially for:
- automatic interaction-memory capture
- minimal-prompt resumability entry behavior
- transport-visible historical recall shaping

## 3. Canonical-versus-derived boundary validation

Add or maintain tests that preserve the intended reading:

- canonical workflow / checkpoint / failure truth remains primary
- interaction memory, file-work metadata, summaries, embeddings, and
  graph-linked structures remain derived support layers
- degraded derived layers should not be described as canonical loss

This is a product-honesty requirement, not only a modeling detail.

## 4. Compatibility-field discipline

Where compatibility aliases remain exposed:
- test them explicitly
- document them explicitly
- avoid silently drifting their meaning

Where compatibility aliases are no longer needed:
- remove them intentionally
- update tests with the same intentionality

---

## Suggested implementation order for validation work

The focused validation work should be established in this order:

1. **resume validation**
   - because it fixes the opening `0.9.0` behavior contract

2. **historical recall validation**
   - because it fixes the supported question class and prevents over-claiming

3. **interaction-memory validation**
   - because boundary-level capture policy should be validated as soon as it is
     designed

4. **file-work metadata validation**
   - because the minimum schema should be stabilized before wider retrieval logic

5. **failure-reuse validation**
   - because it depends partly on the earlier interaction-memory and file-work
     posture being explicit

This ordering is not rigid, but it is the safest bounded path.

- all five validation areas have at least minimum scenario coverage
- response-shaping changes are test-backed where the contract matters
- boundary-layer behavior is tested where `0.9.0` changes behavior

## Relationship to acceptance

This validation plan is not the acceptance checklist itself.

Instead:
- `0.9.0_acceptance_checklist.md` defines **what must be true**
- this plan defines **what focused validation must exist to support those claims**

A milestone should not be treated as accepted merely because implementation
exists.
It should be treated as accepted when:
- the bounded behavior exists
- the bounded behavior is documented
- the bounded behavior is validated strongly enough to trust

## Summary

The bounded `0.9.0` validation frame should ensure the repository can prove:

- minimal-prompt resume behavior is explicit and explainable
- bounded historical progress recall is real and not over-claimed
- interaction memory is captured automatically in a durable bounded way
- prior failures and recoveries are easier to reuse
- file-work metadata is durable and queryable without indexing file contents

That is the focused validation standard for the milestone.
