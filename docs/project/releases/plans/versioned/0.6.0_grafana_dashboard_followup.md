# `0.6.0` Grafana Dashboard Follow-Up

## Purpose

This note defines the bounded `0.6.0` follow-up work needed to bring the current
Grafana dashboards closer to the observability meaning introduced by the
implemented summary hierarchy slice.

It is intended to answer a narrow operational question:

> After `0.6.0` changed `memory_get_context`, summary building, AGE summary
> enrichment, and workflow-completion summary automation, what dashboard changes
> are needed so Grafana reflects the current system behavior instead of only
> older generic workflow/memory counts?

This is **not** a new observability milestone proposal.

It is a focused follow-up note for:

- dashboard semantics
- operational visibility
- `0.6.0` summary hierarchy observability
- bounded dashboard implementation work

---

## Context

The current `0.6.0` slice added or clarified several behaviors that materially
change what operators may want to observe:

- canonical summary persistence through:
  - `memory_summaries`
  - `memory_summary_memberships`
- constrained summary-first retrieval
- direct summary-member memory-item expansion
- bounded graph-backed auxiliary summary-member enrichment
- explicit graph refresh and graph-readiness flows
- workflow-completion summary automation that is:
  - explicit
  - checkpoint-gated
  - non-fatal
- richer workflow-completion payload details including nested `summary_build`
  fields

At the same time, the currently committed Grafana dashboards still mostly show:

- broad workflow counts
- broad memory counts
- failure counts
- generic runtime activity counts

That means the dashboards are still useful, but they do not yet make the
`0.6.0` summary hierarchy slice especially visible.

---

## Current dashboard state

Current dashboard files include:

- `docker/grafana/dashboards/runtime_overview.json`
- `docker/grafana/dashboards/memory_overview.json`
- `docker/grafana/dashboards/failure_overview.json`

### Current strengths

The current dashboards already provide useful baseline visibility for:

- workflow volumes
- attempt/checkpoint/verify-report status distributions
- memory totals
- memory provenance counts
- failure lifecycle counts
- recent workflow activity
- high-level runtime activity timelines

### Current gap relative to `0.6.0`

The current dashboards do **not yet** make the following `0.6.0` outcomes
explicit enough:

- how many canonical summaries now exist
- how many canonical summary memberships now exist
- whether summary building is occurring as expected in the current bounded slice
- whether auto workflow memory is producing summary artifacts, not only memory
  items
- whether graph-backed summary support is being refreshed and treated as derived
- whether the operator should read a low/zero summary state as:
  - expected because no summaries were built yet
  - degraded because automation was not requested
  - concerning because summary creation is unexpectedly absent

This gap matters because `0.6.0` is no longer only about “more memory data.”
It is also about a new bounded summary hierarchy layer.

---

## Why this follow-up is needed

Without a dashboard follow-up, operators looking only at Grafana may still infer
an older pre-`0.6.0` model such as:

- workflows create memory, but summaries are not first-class
- workflow-completion auto-memory is the only notable derived memory effect
- graph-backed summary support is invisible or irrelevant
- there is no practical difference between:
  - memory items
  - canonical summaries
  - derived graph-backed summary enrichment

That would be misleading.

The dashboards do not need to mirror every MCP payload detail, but they should
surface the most important operational consequences of the `0.6.0` slice.

---

## Desired observability outcome

A good bounded `0.6.0` dashboard follow-up should let an operator answer the
following practical questions quickly:

### Canonical summary layer
- How many canonical summaries currently exist?
- How many summary memberships currently exist?
- Are summary rows increasing over time?
- Are summary memberships increasing over time?

### Explicit and automated summary build behavior
- Are workflow-completion auto-memory items being created?
- Are canonical summaries also being created for those workflows when requested?
- Is the summary layer lagging behind auto-memory creation unexpectedly?

### Derived graph support posture
- Is AGE-derived summary support healthy enough for the bounded auxiliary path?
- Is the graph-backed layer being treated as supplementary rather than as
  canonical?
- Are graph-related observability signals visible enough to detect “degraded but
  still relationally correct” operation?

### Operator interpretation
- Can the operator distinguish:
  - no summaries built yet
  - summaries built successfully
  - auto-memory recorded without summary build
  - graph-backed auxiliary support absent or stale

---

## Current signals that should influence dashboard design

The dashboard follow-up should be designed with the current implementation
boundaries in mind.

## 1. Canonical summary truth is relational

The dashboard should reinforce the current `0.6.0` reading that:

- `memory_summaries`
- `memory_summary_memberships`

are canonical relational state.

This means a dashboard panel showing summary counts is not just “extra derived
stats.”
It is now an important part of the actual bounded product slice.

## 2. Graph-backed summary support is derived

The dashboard should not imply that graph-backed summary support is the only
correctness path.

If graph-related summary support is surfaced, it should be read as:

- additive
- derived
- degradable
- useful for observability and auxiliary enrichment
- not canonical truth

## 3. Workflow summary automation is gated

Dashboard wording should reflect that current workflow-summary automation is not
always-on.

The implementation currently uses a checkpoint-gated request posture, reflected
by policy fields such as:

- `default_requested = false`
- `request_field = latest_checkpoint.checkpoint_json.build_episode_summary`
- `trigger = latest_checkpoint.build_episode_summary_true`

The dashboard should therefore avoid implying that every workflow completion
should automatically result in a summary.

---

## Recommended dashboard changes

## 1. Update `memory_overview.json`

This is the best current place to add most of the bounded `0.6.0` summary-layer
visibility.

### Recommended new stat panels

Add stat panels for:

- canonical summary count
- canonical summary membership count
- workflow-completion auto-memory item count
- auto-memory summaries count, if an observability view can support it cleanly

### Recommended new table or breakdown panels

Add a table or pie/table breakdown for:

- summary kinds
- summary builder/source metadata if already available through stable
  observability views
- summary-to-membership ratios or a simple membership density reading

### Recommended new timeline panels

If observability SQL views support it cleanly, add a timeline for:

- summary creation events
- summary membership creation events

This should be kept bounded and should not require introducing a large new
event-sourcing model just for Grafana.

### Why this belongs in `memory_overview`

The `0.6.0` slice added a new canonical memory layer.
That belongs naturally in the memory dashboard rather than being hidden behind
workflow-only or runtime-only aggregates.

---

## 2. Update `runtime_overview.json`

This dashboard is the best place to expose bounded policy and runtime-status
interpretation.

### Recommended additions

Add panels or tables that surface:

- current summary automation policy reading
- whether the current policy is checkpoint-gated
- whether graph-derived summary support is available/ready if a stable SQL-backed
  projection exists
- whether the current operator reading should be:
  - relationally healthy
  - graph-ready
  - degraded but relationally correct

### Important design constraint

Do **not** attempt to mirror ephemeral HTTP `/debug/runtime` JSON directly into
Grafana unless there is already a stable persistence or observability-view path
for it.

Grafana in this repository should continue to read from stable PostgreSQL-backed
observability state, not from ad hoc runtime JSON scraping.

That means the runtime dashboard follow-up should prefer:

- observability SQL views
- stable database-backed summary/automation counters or policy indicators

rather than a direct “display the debug route payload” approach.

---

## 3. Keep `failure_overview.json` mostly unchanged

The failure dashboard is already conceptually focused.

Only update it if the `0.6.0` summary hierarchy work has introduced clear new
failure categories or operational diagnostics that are already represented in the
observability schema.

Do **not** force summary-specific panels into the failure dashboard unless the
failure model itself has meaningfully changed.

This keeps the slice semantically small.

---

## Recommended implementation shape

## 1. Prefer observability-view additions before dashboard overreach

If the existing dashboards cannot cleanly express `0.6.0` summary-layer state
from current SQL views, the first implementation step should be:

- add or extend stable observability SQL views

Possible needed view additions or extensions may include counts for:

- summary rows
- summary memberships
- summary kinds
- workflow-completion auto-memory episodes/items versus built summaries
- latest summary activity timestamps

The dashboard should then read from those stable views.

## 2. Keep the first dashboard slice small

A good bounded first dashboard follow-up would be:

### Memory dashboard
- add summary count
- add summary membership count
- add summary-related latest activity markers
- add one summary-kind or summary-source breakdown if cleanly supported

### Runtime dashboard
- add one bounded summary-automation policy/readiness panel if backed by stable
  observability SQL

That is enough to make the `0.6.0` slice visible without inventing a whole new
dashboard family.

---

## Proposed concrete operator questions to support

After this follow-up, an operator should be able to answer:

1. Did the repository create canonical summaries at all?
2. Are summary memberships being written along with summaries?
3. Did workflow-completion auto-memory create memory items only, or summaries
   too?
4. Is the current graph-backed summary support healthy, degraded, or not in use?
5. Is the current summary layer behaving like a real bounded product slice, not
   just hidden background metadata?

If the dashboard changes do not help answer those questions, they are probably
too weak for the current milestone.

---

## What this follow-up should not do

This Grafana follow-up should **not**:

- attempt a broad observability redesign
- scrape MCP payloads directly into Grafana
- imply graph-backed summary support is canonical truth
- imply workflow summary automation is always-on
- require a full new metrics subsystem just to show `0.6.0` progress
- mix unrelated failure-observability concerns into the memory dashboard
- overfit the dashboard to every single nested `auto_memory_details` field

The point is to improve release-appropriate observability for the current
implemented slice, not to build a whole new telemetry platform.

---

## Relationship to MCP payload evolution

This follow-up is related to the `0.6.0` MCP payload changes, but it should not
try to duplicate them exactly.

### MCP payloads now expose:
- `summary_selection_kind`
- retrieval route metadata
- `memory_context_groups`
- nested `auto_memory_details.summary_build`

### Grafana should instead expose:
- persistent or aggregate operational signals derived from the same bounded slice
- summary counts
- membership counts
- summary-build visibility where stable
- policy/readiness interpretation where stable

In other words:

- MCP payloads explain individual request outcomes
- Grafana should explain ongoing operational state

That distinction should remain clear.

## Acceptance criteria for this follow-up

This follow-up should be considered complete when:

1. the dashboard set reflects canonical summary-layer visibility strongly enough
   for `0.6.0`
2. dashboard wording does not blur relational truth with derived graph support
3. dashboard wording does not blur auto-memory recording with summary-build
   success
4. the first bounded implementation remains semantically small
5. any required observability SQL/view changes are documented and reflected in
   dashboard queries
6. the resulting Grafana panels make at least the following visible:
   - canonical summary count
   - canonical summary membership count
   - workflow-completion auto-memory relationship to summary creation, if stable
     enough to expose
7. the dashboard changes are validated in the same bounded, behavior-preserving
   spirit as the rest of the `0.6.0` slice

## Recommended next step

Implement this follow-up as a small observability slice:

1. define the minimal stable observability signals needed for summary-layer
   visibility
2. update the Grafana dashboard JSONs to surface those signals
3. keep the first slice focused on memory and runtime overview
4. validate that the resulting dashboard now helps operators understand the
   actual `0.6.0` bounded summary hierarchy behavior

Once that is done, treat this follow-up as closed unless later milestones expand
summary observability further.
