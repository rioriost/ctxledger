# `0.9.0` Acceptance Review

## Purpose

This document records a release-facing acceptance assessment for the bounded
`0.9.0` strengthening slice implemented in `ctxledger`.

It is intended to answer a practical release question:

> Does the current repository satisfy the documented `0.9.0` acceptance shape
> strongly enough to be treated as accepted for its intended bounded scope?

This review is not a new roadmap proposal.

It evaluates the current repository reading against:

- `docs/project/releases/plans/versioned/0.9.0_acceptance_checklist.md`
- `docs/project/releases/plans/versioned/0.9.0_ctxledger_rules_strengthening_plan.md`
- `docs/project/releases/plans/versioned/0.9.0_focused_validation_plan.md`
- `docs/project/releases/0.9.0_acceptance_inventory.md`

and the bounded contract docs for:

- minimal-prompt resumability
- historical progress recall
- interaction memory
- interaction capture boundaries
- file-work metadata
- failure reuse

This review also records the final verification reading available at acceptance
+review time:

- targeted focused validation for the implemented `0.9.0` slice has been run and
  recorded during implementation
- the release-facing artifacts now treat that targeted validation as the current
  authoritative verification baseline for the accepted bounded slice
- broader future follow-up validation remains separate from the acceptance
  judgment for the already-implemented bounded scope

---

## Acceptance scope

This review is specifically about the bounded `0.9.0` milestone direction now
described by the repository.

That bounded direction is:

- minimal-prompt resumability from workspace context
- bounded historical progress recall
- automatic interaction-memory capture
- stronger failure-pattern reuse and avoidance
- durable file-work metadata capture without file-content indexing
- tighter operational alignment between `ctxledger` and repository `.rules`

This review does **not** claim completion of:

- broad graph-memory redesign
- unconstrained conversational historical QA
- broad file-content indexing
- full graph-linking closeout for every interaction-memory path
- a fully closed repeated-failure classification system beyond the bounded
  current slice

It assesses whether the current repository is strong enough to be treated as
accepted for the implemented bounded `0.9.0` scope.

---

## Review result

**Acceptance result:** accepted with bounded-scope caveats  
**Acceptance reading:** accepted as a **bounded `0.9.0` strengthening milestone
slice**  
**Review status:** passed with explicit partial and deferred follow-up candidates
preserved

The current repository supports the following bounded reading:

- minimal-prompt resume behavior is now contractually explicit and materially
  more explainable than before
- bounded historical recall is now documented and shaped more clearly
- user requests and agent responses are automatically captured as durable
  interaction memory through bounded HTTP and MCP paths
- interaction memory is distinguishable and retrieval-ready
- file-work metadata is durably captured, queryable, and visible in explanation
  surfaces without requiring Git-managed file-content indexing
- failure-reuse explanation is materially stronger because interaction memory and
  file-work metadata now contribute directly to bounded search-side reasoning
- grouped interaction-memory context output now exists in `memory_get_context`
- operator-facing observability now includes interaction and file-work memory
  counts in the current bounded stats surfaces

This means the repository can reasonably treat `0.9.0` as **accepted for the
implemented bounded scope**, while still preserving a set of explicit partial and
deferred follow-up items.

**Final verification reading:** the current acceptance judgment is backed by the
focused validation that has been executed for the implemented slice, including:

- CLI observability coverage for interaction/file-work metrics
- PostgreSQL-backed interaction-group context validation
- search-side interaction/file-work/failure-reuse explanation validation
- MCP and HTTP interaction capture validation
- serialization validation for grouped interaction-memory output

This does not widen the claim into “every imaginable future follow-up is already
validated”.
It means the implemented bounded slice now has enough verification evidence to
support acceptance honestly.

## Acceptance criteria used

The practical acceptance baseline for this review is:

1. the repository has an explicit bounded contract for minimal-prompt resume
2. representative workspace-context resume behavior is explainable and testable
3. historical progress recall is explicitly bounded and explainable
4. interaction memory is captured automatically rather than depending only on
   manual memory calls
5. interaction memory is distinguishable and retrieval-ready
6. file-work metadata is durably captured and queryable without file-content
   indexing
7. file-work metadata contributes to failure-reuse behavior in bounded
   representative cases
8. `.rules` and implementation still preserve canonical-first posture
9. operator-facing observability remains useful and canonical-first
10. the implemented slice is backed by focused validation and aligned docs

## Detailed review

## 1. Minimal-prompt resumability

### Expected

A user should be able to say:

- `resume`
- `continue`
- `作業を再開`

and the repository should provide a bounded, explainable, workspace-context
continuation-selection model rather than depending mainly on ad hoc local notes.

### Evidence

The repository now has an explicit bounded contract document:

- `docs/memory/design/minimal_prompt_resume_contract.md`

The current slice explicitly preserves:

- workflow-id-specific resume versus workspace-context resume
- selected-versus-latest explanation
- mainline-versus-detour explanation
- canonical-first workflow/checkpoint boundary

Focused validation now covers representative selection behavior such as:

- running workflow preference
- latest non-terminal preference when appropriate
- non-terminal preference over latest terminal
- non-detour preference over latest detour-like candidate
- explicit selected-candidate explanation visibility

### Result

**Pass**

Minimal-prompt resume is now strong enough to be treated as a real bounded
`0.9.0` milestone behavior.

### Caveat

Prior-mainline recovery should still be read as bounded and partially open for
future strengthening rather than as fully exhausted design space.

---

## 2. Resume reasoning remains canonical-first

### Expected

Resume behavior should remain grounded in canonical workflow/checkpoint state and
must not become a second workflow-truth system.

### Evidence

The current contract docs explicitly preserve the canonical/derived boundary.

Current runtime behavior still treats:

- selection
- ranking
- mainline/detour interpretation
- selected-vs-latest explanation

as derived operational reading over canonical workflow and checkpoint truth.

Interaction memory does not currently redefine canonical workflow state.

### Result

**Pass**

The current `0.9.0` resume posture remains canonical-first.

---

## 3. Bounded historical progress recall

### Expected

The repository should support a bounded class of historical questions such as:

- what was completed yesterday?
- what happened for this keyword?
- where did we stop?
- what remains?
- what did the user ask the agent about this topic yesterday?

without implying unconstrained historical omniscience.

### Evidence

The repository now has an explicit bounded historical recall contract:

- `docs/memory/design/historical_progress_query_contract.md`

That contract explicitly covers:

- time anchors
- keyword/topic anchors
- workflow/task anchors
- file-work anchors
- failure/recovery anchors

and explicitly bounds out:

- unconstrained historical QA
- broad repository analytics
- broad file-content indexing

Current context-shaping behavior still exposes explicit retrieval and explanation
surfaces, and grouped interaction-memory output now strengthens historical
explainability rather than weakening it.

### Result

**Pass**

Bounded historical recall is now clear enough to be accepted for the milestone’s
intended scope.

### Caveat

The milestone should still avoid claiming broad conversational historical QA or
fully mature end-to-end handling for every future historical question class.

---

## 4. Historical answer assembly is explainable

### Expected

Historical recall should not only work; it should explain how the answer was
assembled.

### Evidence

The current repository preserves:

- grouped primary output
- route-aware details
- interaction-memory visibility
- file-work metadata visibility
- bounded failure-reuse explanation in search surfaces

The docs also explicitly require explainability as part of the contract.

### Result

**Pass**

Historical answer assembly is explainable enough for the bounded `0.9.0` slice.

## 5. Automatic interaction-memory capture

### Expected

User requests and agent responses should become durable interaction memory
through an automatic path rather than depending mainly on discretionary manual
capture.

### Evidence

The repository now has both contract and implementation support for this:

Docs:

- `docs/memory/design/interaction_memory_contract.md`
- `docs/memory/design/interaction_capture_boundary_design.md`

Implementation:

- bounded HTTP runtime request/response persistence
- bounded MCP RPC request/response persistence
- interaction memory stored with:
  - `type`
    - `interaction_request`
    - `interaction_response`
  - `provenance`
    - `interaction`
  - interaction metadata such as:
    - `interaction_role`
    - `interaction_kind`

Focused validation now covers request-side and response-side capture through both
HTTP and MCP paths.

### Result

**Pass**

The repository now automatically records user requests and agent responses as
durable interaction memory for the bounded current slice.

---

## 6. Interaction memory is distinguishable and retrieval-ready

### Expected

Interaction memory should be visibly distinct from other memory classes and
should participate in bounded retrieval rather than only being stored.

### Evidence

Interaction memory is now distinguishable by:

- `type`
- `provenance`
- interaction metadata

It is also retrieval-ready through:

- `memory_search`
- `memory_get_context`

Current behavior includes:

- interaction-aware ranking preference in bounded search cases
- grouped interaction-memory output in `memory_get_context`
- visible interaction role/kind details
- visible file-work metadata when present

### Result

**Pass**

Interaction memory is now retrieval-ready strongly enough for the current
bounded milestone reading.

### Caveat

Broader graph-linking / relation-writing closeout for interaction memory remains
a partial future-strengthening area rather than a completed acceptance claim.

## 7. Failure-pattern reuse and avoidance

### Expected

The repository should make prior failure and recovery knowledge easier to
surface, especially when interaction memory and file-work metadata provide useful
bounded signals.

### Evidence

The repository now has an explicit failure reuse contract:

- `docs/memory/design/failure_reuse_contract.md`

The current implementation strengthens failure reuse through:

- file-work-aware search-side explanation
- interaction-aware search-side explanation
- bounded `failure_reuse_detail`
- bounded search ranking reason:
  - `failure_reuse_file_work_signal`

This is materially better than earlier slices where failure reuse was largely an
aspirational reading.

### Result

**Pass**

The current repository now surfaces bounded failure-reuse behavior strongly
enough to count for the implemented slice.

### Caveat

A fully mature fresh-vs-repeated-known-failure classification system is still not
closed out.
That part should remain bounded and partial rather than over-claimed.
