# `0.7.0` Release-Ready Behavior Summary

## Purpose

This document records the current release-facing behavior reading for the bounded
`0.7.0` task-recall slice implemented in `ctxledger`.

It is intended to answer a practical release question:

> What does the currently implemented `0.7.0` slice actually do, what remains
> intentionally bounded, and what still appears to be deferred or incomplete?

This is a **behavior summary**, not a full acceptance review.

It exists to give operators, maintainers, and future planning work a clear
release-facing reading of the current repository state without overstating the
milestone as fully closed.

---

## Scope

This summary is specifically about the bounded `0.7.0` task-recall slice
described in:

- `docs/project/releases/plans/versioned/task_recall_0_7_0_plan.md`
- `docs/project/product/roadmap.md`
- `docs/project/releases/CHANGELOG.md`
- `docs/project/releases/0.7.0_status_review.md`

This summary does **not** claim that all longer-term task-thread, return-target,
or concept-routing ambitions are complete.

It summarizes the currently implemented bounded slice.

---

## Current release-facing reading

The current repository state supports the following reading:

- `0.7.0` has implemented a substantial bounded task-recall slice
- the repository can now surface richer continuation-selection details than in
  earlier milestones
- the main current behavior improvement is not only ranking, but also
  explanation-rich continuation inspection
- `memory_get_context` and `memory_search` now expose materially stronger
  task-recall detail surfaces for bounded workspace-scoped retrieval
- the current slice still remains derived-on-top-of-canonical-state rather than
  introducing a new canonical workflow-truth layer
- the milestone is more implemented than older planning shorthand implied
- but the release line is still not fully closed from a closeout/governance
  perspective

A practical shorthand is:

- implemented as a **bounded task-recall milestone slice**
- not yet documented as a fully accepted or fully closed milestone

---

## What `0.7.0` implemented

## 1. Objective-aware continuation ranking

The current `0.7.0` slice now supports task-recall ranking that can prefer
stronger mainline evidence over simple freshness alone.

Current implemented signals include:

- latest checkpoint current objective
- latest checkpoint next intended action
- latest attempt presence
- latest checkpoint presence
- workflow terminal versus non-terminal status
- detour-like wording penalties from ticket and checkpoint text

This means recent side work is less likely to eclipse the real continuation
target when stronger explicit mainline signals exist.

---

## 2. Richer `memory_get_context` task-recall details

`memory_get_context` now has a materially richer task-recall explanation
surface.

Current bounded details include:

- selected workflow identity
- latest considered workflow identity
- running workflow identity where applicable
- return-target presence, workflow id, basis, and source
- primary-objective presence, text, and source
- task-thread presence, basis, and source
- selected checkpoint step and summary
- latest considered checkpoint step and summary
- selected-versus-latest comparison details
- comparison explanations and summary explanations
- ranking details and explanation details
- selected and latest detour-like classification details
- selected workflow terminal details

This is a meaningful change from earlier, simpler freshness-oriented retrieval.

---

## 3. Explicit latest-versus-selected comparison behavior

The current bounded slice now clearly distinguishes:

- the latest considered workflow
- the selected continuation target

This distinction is surfaced through:

- candidate comparison details
- checkpoint comparison details
- detour classification comparison details
- resumability-oriented signal comparison details
- explanation lists describing why the selected continuation target differed from
  the latest considered candidate

This allows operators and agents to inspect not only the result, but also the
selection reasoning.

---

## 4. Return-target explanation surface

The current `0.7.0` slice now exposes an explicit bounded return-target reading.

Current return-target behavior includes:

- return-target presence
- return-target workflow identity
- return-target basis
- return-target source

The current basis/source reading can distinguish cases such as:

- checkpoint current objective
- checkpoint next intended action
- ranking-based candidate selection
- detour override
- terminal override
- latest candidate retention

This is currently a **derived explanation surface**, not a first-class persisted
canonical entity.

---

## 5. Primary-objective detail surface

The current bounded slice now surfaces explicit primary-objective details when
checkpoint evidence supports them.

Current bounded behavior includes:

- primary-objective presence
- primary-objective text
- primary-objective source

This makes the retrieval surface more capable of answering questions such as:

- what is the current main objective?
- what should be re-foregrounded after side work?

without requiring a separate canonical objective model.

---

## 6. Task-thread explanation surface

The current `0.7.0` slice now includes a bounded derived task-thread detail
surface.

Current bounded behavior includes:

- task-thread presence
- task-thread basis
- task-thread source

This should be read as an explanation-oriented continuity surface, not as a
complete persisted task-thread model.

---

## 7. Prior-mainline recovery before latest detour

The current repository reading supports a bounded prior-mainline recovery path in
`memory_get_context`.

Current bounded behavior includes:

- latest detour candidate details
- prior-mainline presence
- prior-mainline workflow identity
- prior-mainline candidate details

This means the current slice is already capable of recovering a prior mainline
candidate before a latest detour-like workflow in bounded cases.

That is materially stronger than a pure “latest wins” model.

---

## 8. Latest-detour versus selected-continuation separation

The current bounded slice also preserves the distinction between:

- the latest detour-like candidate
- the selected continuation target

This separation is visible in:

- `memory_get_context` detail payloads
- candidate comparison surfaces
- explanation summaries
- workspace-resume-oriented reasoning

This means the system can now say, in effect:

- “this was the most recent side-task-like work”
- “this is still the better continuation target”

rather than collapsing those into a single freshness answer.

---

## 9. Bounded `memory_search` task-recall bridge

`memory_search` now includes a bounded task-recall bridge for workspace-scoped
searches.

Current bounded search behavior includes:

- latest considered workflow identity
- selected continuation workflow identity
- latest-versus-selected candidate comparison details
- comparison-summary explanations in divergent multi-candidate contexts
- a small selected-continuation-target bonus in bounded divergent contexts

This should be read as:

- concept-to-task recovery scaffolding
- derived explanation and ranking-support behavior
- not a second workflow-truth mechanism

---

## 10. Transport and serialization support

The current bounded slice also appears to preserve these task-recall details
through the main user-facing bounded surfaces.

Current repository reading supports task-recall field continuity across:

- memory service responses
- serializer defaults and compatibility aliases
- MCP-facing tool responses
- HTTP-facing bounded transport behavior
- workspace-resume-facing reasoning payloads

This matters because the milestone is only useful if its explanation surfaces are
actually visible to calling agents and operators.

---

## What `0.7.0` intentionally does **not** yet claim

The current bounded slice should **not** be read as having already delivered:

- a complete task-thread model
- a first-class persisted return-target entity
- broad concept-to-task routing beyond the current bounded
  `memory_get_context` and `memory_search` path
- a canonical task-thread record replacing current workflow/checkpoint truth
- a replacement of recency/resumability signals with objective-aware signals;
  the current slice augments them instead
- a broad architectural redesign of workflow/memory truth ownership

These remain important boundary conditions for interpreting the release
correctly.

## Canonical boundary reading

The current `0.7.0` slice should still be read under the existing canonical
boundary:

- PostgreSQL-backed workflow and checkpoint state remain canonical
- task-recall selection is a derived retrieval-and-explanation layer
- return-target fields are currently derived
- task-thread fields are currently derived
- `memory_search` task-recall context is currently derived
- no new second workflow-truth system has been introduced

This is an important release-facing constraint.

The milestone improves recoverability and inspection without changing the
canonical system-of-record posture.

- objective-aware continuation ranking

- richer continuation explanation details
- bounded prior-mainline recovery
