# `0.8.0` Closeout Note

## Purpose

This note records the final release-facing closeout reading for the bounded
`0.8.0` remember-path milestone in `ctxledger`.

It is intended to answer a practical closeout question:

> Should `0.8.0` now be treated as closed, and if so, under what exact scope and
> caveats?

This note is deliberately narrower than a broad roadmap narrative and more final
than an implementation-progress or planning document.

---

## Closeout result

**Closeout result:** closed  
**Closeout scope:** closed as a **bounded `0.8.0` remember-path milestone slice**  
**Closeout posture:** accepted with explicit scope boundaries and bounded
follow-up candidates

The current repository should now treat `0.8.0` as closed for its intended
bounded scope.

That bounded scope is:

- stronger remember-path recording quality
- clearer checkpoint-origin versus completion-origin memory behavior
- a more explicit primary grouped `memory_get_context` reading
- a bounded `primary_only` client path for grouped-primary consumers
- bounded ranking improvement for completion-vs-checkpoint lexical ties
- clearer AGE readiness interpretation and remediation guidance
- operator-facing canonical-summary and derived-graph posture metrics
- safer local small-stack posture for ordinary development use

This closeout does **not** mean every larger future memory, ranking, graph, or
compatibility-retirement idea is complete.

It means the repository now has enough implemented behavior, operator-facing
documentation, acceptance framing, and validation coverage to close the bounded
milestone honestly.

---

## What is being closed

`0.8.0` is closed for the following implemented and documented bounded outcome:

1. **Remember-path recording is more operationally meaningful**
   - the repository no longer reads only as sparse episodic capture plus
     optional extras
   - checkpoint-origin and completion-origin memory recording are visible enough
     to matter operationally

2. **Checkpoint-origin and completion-origin memory are distinguished**
   - `workflow_checkpoint_auto`
   - `workflow_complete_auto`
   are now visibly distinct in the current remember-path reading

3. **`memory_get_context` now has a stronger primary grouped reading**
   - `memory_context_groups` is the primary grouped hierarchy-aware surface
   - route and selection metadata in `details` support that grouped reading
   - flatter compatibility and convenience fields remain available where
     relevant, but are no longer the strongest contract reading

4. **A bounded `primary_only` client path now exists**
   - `primary_only = true` narrows the response shape toward the grouped primary
     surface
   - this is a shaping mode over the same retrieval behavior, not a separate
     retrieval algorithm

5. **Search ranking now includes a bounded completion-over-checkpoint tie-break**
   - when lexical evidence ties, completion-origin memory can outrank
     checkpoint-origin memory
   - this behavior is explained through an explicit ranking reason rather than
     remaining implicit

6. **AGE readiness is now more legible**
   - readiness is no longer only a simple ready/unavailable reading
   - the current operator-facing posture now distinguishes:
     - `ready`
     - `stale`
     - `degraded`
     - `unknown`

7. **Operator-facing stats now expose defined summary and graph-posture metrics**
   - canonical summary volume:
     - `memory_summary_count`
     - `memory_summary_membership_count`
   - derived graph posture:
     - `age_summary_graph_ready_count`
     - `age_summary_graph_stale_count`
     - `age_summary_graph_degraded_count`
     - `age_summary_graph_unknown_count`

8. **Local small-stack posture is safer**
   - ordinary local development is no longer framed as requiring an external
     embedding credential path
   - the repository preserves ordinary `.env`-based local operator reading while
     keeping external embedding providers opt-in
