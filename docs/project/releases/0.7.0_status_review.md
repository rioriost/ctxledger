# `0.7.0` Status Review

## Purpose

This document records a release-facing status reading for the current `0.7.0`
task-recall milestone.

It has two goals:

1. turn the remaining `0.7.0` work into a TODO checklist that can be copied
   directly into implementation issues
2. assess, from the current documented repository reading, whether each item
   appears:
   - implemented
   - partially implemented
   - still missing
   - or at risk of being stale documentation

This is **not** an acceptance review.

Unlike `0.6.0`, the current `0.7.0` reading is still a status-and-gap artifact,
not a closeout statement.

---

## Current reading

The main repository docs consistently describe `0.7.0` as an **in-progress
task-recall milestone** rather than a fully closed milestone.

The strongest current status statement appears in:

- `docs/project/releases/plans/versioned/task_recall_0_7_0_plan.md`

That plan explicitly says the current slice should be read as:

- an incremental `0.7.0` retrieval improvement
- not full milestone closeout
- roughly `70%` complete
- not yet release-ready

The roadmap and changelog align with that reading.

As a result, the practical current repository reading is:

- the first meaningful `0.7.0` task-recall slice is real
- the explanation and comparison surface is materially richer than before
- bounded concept-to-task retrieval is already present
- but several closeout expectations still remain open

## High-level status summary

### What appears clearly implemented already

- objective-aware task-recall continuation ranking
- stronger latest-versus-selected continuation comparison details
- bounded `memory_search` task-recall integration for workspace-scoped search
- return-target detail surfacing
- primary-objective detail surfacing
- task-thread detail surfacing
- latest detour candidate versus selected continuation target separation in
  `memory_get_context`
- prior-mainline candidate recovery support in `memory_get_context`
- substantial test coverage around task-recall details, search comparison
  details, and workspace-resume heuristics

### What still appears incomplete or at least not fully closed

- broader concept-to-task routing beyond the current bounded
  `memory_get_context` and `memory_search` path
- broader integration of task-recall semantics into adjacent retrieval surfaces
  where justified
- maintainability hardening for the oversized task-recall-related split stream
  is now complete:
  - the first oversized runtime task-recall source files have been split
  - the oversized memory-context visibility test file has now also been split
  - the oversized memory-context grouping test file has now also been split
  - the oversized memory-context task-recall test file has now also been split
  - the final oversized memory-context grouping-summary-routes test file has now
    also been split

### What looks potentially stale in the docs

This stale-docs risk is now narrower than before.

The earlier planning posture understated implemented work in areas such as:

- stronger recovery of the previous primary workflow before the latest detour
- better preservation and surfacing of latest detour versus selected
  continuation target as distinct operational concepts

Those areas have now been refreshed in the main `0.7.0` planning/status docs.

The remaining likely stale area is maintainability hardening language that still
reads as if oversized source/test splitting were only planned, even though the
current repository now includes the first semantic splits for:

- runtime task-recall source files
- runtime task-recall test files
- memory-context visibility test files

That does **not** mean the hardening stream is finished.
It means the status reading should distinguish between:

- hardening started and partially completed
- hardening fully closed
