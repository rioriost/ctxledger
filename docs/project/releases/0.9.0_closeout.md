# `0.9.0` Closeout Note

## Purpose

This note records the release-facing closeout reading for the bounded `0.9.0`
strengthening milestone in `ctxledger`.

It is intended to answer a practical closeout question:

> Should `0.9.0` now be treated as closed, and if so, under what exact scope,
> limits, and follow-up posture?

This note is narrower than roadmap prose and more final than implementation
inventory or planning notes.

It should be read as the repository’s bounded closeout statement for the
current `0.9.0` slice.

---

## Closeout result

**Closeout result:** closed  
**Closeout scope:** closed as a **bounded `0.9.0` strengthening milestone slice**  
**Closeout posture:** accepted with explicit boundaries and explicit follow-up
candidates

The current repository should now treat `0.9.0` as closed for its intended
bounded scope.

That bounded scope is:

- stronger minimal-prompt resume contract definition
- bounded historical progress recall contract definition
- automatic interaction-memory capture at bounded MCP / HTTP interaction
  boundaries
- retrieval-ready interaction-memory visibility in search and context output
- durable file-work metadata capture without broad file-content indexing
- stronger bounded failure-reuse explanation through interaction and file-work
  metadata
- explicit grouped interaction-memory output in `memory_get_context`
- stronger release-facing documentation and acceptance inventory for the
  milestone
- initial operator-facing observability for interaction-memory and file-work
  memory counts

This closeout does **not** mean every larger future historical QA, failure
reuse, graph-linking, or observability ambition is complete.

It means the repository now has enough implemented behavior, bounded contract
docs, release-facing framing, and focused validation coverage to close the
current intended `0.9.0` slice honestly.

---

## What is being closed

`0.9.0` is closed for the following implemented and documented bounded outcome.

### 1. Minimal-prompt resume is now a real bounded contract

The repository no longer treats minimal-prompt resume as only an informal
aspiration.

It now has:

- an explicit contract for workspace-context continuation selection
- explicit selected-versus-latest reading
- explicit mainline-versus-detour reading
- preserved canonical-first workflow truth boundaries
- focused tests for representative selection cases

This means the repository can now read `resume`, `continue`, and similar prompts
through a bounded, explainable continuation model rather than only through
ad hoc interpretation.

### 2. Historical progress recall is now explicitly bounded

The repository now has a documented bounded historical-question contract.

That contract explicitly supports a practical class of questions such as:

- what was completed yesterday?
- what happened for this keyword?
- where did we stop?
- what remains?
- what did the user ask the agent about this topic?

It also explicitly bounds out:

- unconstrained historical QA
- broad repository analytics
- broad file-content indexing
- broad graph-first truth claims

This means the repository now has a clearer product-honest historical recall
posture rather than only scattered retrieval capability.

### 3. Interaction memory is now automatically captured

The repository now automatically records bounded interaction memory for:

- request-side MCP tool calls
- response-side MCP tool calls
- request-side MCP resource reads
- response-side MCP resource reads
- request-side HTTP runtime tool dispatch
- response-side HTTP runtime tool dispatch
- request-side HTTP runtime resource reads
- response-side HTTP runtime resource reads

Interaction memory is now durably represented as memory items with explicit
interaction identity through:

- `type`
  - `interaction_request`
  - `interaction_response`
- `provenance`
  - `interaction`

This means user requests and agent responses are no longer treated only as
ephemeral transport state for the current bounded slice.

### 4. Interaction memory is now retrieval-ready

Interaction memory is now more visible in the retrieval surfaces that matter for
the milestone.

In particular:

- `memory_search` now supports bounded interaction-aware and file-work-aware narrowing and ranking
- `memory_get_context` now supports explicit grouped interaction-memory output as the primary grouped context-reading surface
- interaction role and kind are visible in current retrieval/explanation
  surfaces
- interaction memory remains canonical-second rather than becoming a competing
  workflow-truth layer

This means the repository now has a stronger claim that interaction memory is
not only stored, but actually usable later.

### 5. File-work metadata is now durably queryable

The repository now supports bounded durable file-work metadata such as:

- `file_name`
- `file_path`
- `file_operation`
  - `create`
  - `modify`
- `purpose`

This metadata is now visible through:

- interaction-memory persistence
- bounded search filtering
- grouped context output
- failure-reuse explanation surfaces

The milestone preserves an explicit boundary:

- Git-managed file contents themselves do **not** need to be indexed as memory
  search targets for `0.9.0`

This is a core part of the closeout reading.

### 6. Failure reuse is now stronger through interaction/file-work explanation

The current `0.9.0` slice does not claim a fully general repeated-failure
reasoning system.

However, it does now provide stronger bounded failure-reuse explanation through:

- interaction metadata
- file-work metadata
- search-side failure-reuse detail shaping
- bounded ranking reasons such as:
  - `failure_reuse_file_work_signal`

This means the repository now has a more operational failure-reuse posture than
it had before this slice, even though broader repeated-failure interpretation
remains a follow-up area.

### 7. `memory_get_context` now has explicit grouped interaction output

The repository now emits explicit grouped interaction-memory output in
`memory_get_context`.

This grouped output can now:

- separate interaction memory from non-interaction memory
- surface grouped interaction counts
- surface interaction roles and kinds present
- surface file-work-memory presence and counts

The grouped interaction output is now supported by both:
- serialization-level validation
- PostgreSQL-backed integration validation

This is one of the strongest closeout signals in the current slice because it
moves interaction memory from “stored somewhere” to “visible in a structured,
client-usable recall surface”.

### 8. Operator-facing observability now includes interaction/file-work memory counts

The current operator-facing stats surfaces now include bounded interaction/file-
work observability through:

- `interaction_memory_item_count`
- `file_work_memory_item_count`

These appear in:

- `stats`
- `memory-stats`

These should be read as operator signals rather than canonical truth.
This does not yet amount to a complete interaction/file-work observability
program, but it is enough to close the current bounded observability addition
for the slice.
