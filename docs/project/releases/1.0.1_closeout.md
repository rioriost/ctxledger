# `1.0.1` Closeout Note

## 1. Purpose

This note closes out the bounded `1.0.1` stabilization slice for `ctxledger`.

`1.0.1` should be read as an **operations and observability hardening patch**
that follows the completed `1.0.0` Azure milestone, not as the Azure release
itself and not as a new architectural milestone.

The main purpose of this slice was to improve:

- restart-time reliability for the derived AGE-backed summary layer
- natural file-work recording during bounded file-touching work
- operator clarity in CLI and Grafana observability
- live small-auth validation for the intended runtime behavior
- explicit interpretation of derived-layer state instead of ambiguous `No data`

---

## 2. Scope

The bounded `1.0.1` scope covered five closely related concerns after the
`1.0.0` Azure release:

1. automatic refresh behavior for the derived AGE summary graph
2. natural `file_work_record` production in bounded file-touching flows
3. live validation of the small-auth Docker deployment path
4. repair of live observability gaps discovered during validation
5. replacement of ambiguous derived-memory observability readings with explicit
   state and reason fields

This slice intentionally did **not** expand the canonical truth boundary.

Canonical truth remains in PostgreSQL for:

- workflows
- attempts
- checkpoints
- verify reports
- episodes
- memory items
- canonical summaries
- canonical summary memberships

Derived graph state, rankings, and other support layers remain subordinate and
degradable.

---

## 3. Closed outcomes

## 3.1 AGE startup refresh automation

The derived AGE summary graph now has a shared refresh path and a startup-time
automatic repair path for bounded degraded states.

Closed behavior:

- startup can trigger refresh when derived graph state is:
  - `graph_unavailable`
  - `graph_stale`
  - `graph_read_failed`
- operator fallback remains available through the explicit refresh command
- startup and operator refresh now follow the same rebuild logic rather than
  separate drifting implementations

This matters because restart-time derived-graph thinness should not require a
manual repair loop every time the canonical summary state is already present.

## 3.2 Shared AGE refresh reliability fixes

During live validation, the restart-time AGE path exposed two concrete issues.

Those issues were fixed in the `1.0.1` slice:

- the shared AGE refresh helper had incorrect f-string escaping for Cypher
  property maps, which caused a runtime `NameError`
- the default small-auth startup path could abort before the server came up when
  Grafana database credentials were not configured

Closed result:

- the AGE refresh helper now rebuilds summary nodes and `summarizes` edges
  correctly
- the small-auth startup flow no longer blocks server startup just because the
  Grafana observability helper cannot run with missing credentials
- live readiness can now return:
  - `graph_ready`
  - `readiness_state = ready`
  - `operator_action = no_action_required`

## 3.3 Natural file-work recording

The bounded file-work posture was strengthened so that file-touching work can
leave a durable trail more naturally.

Closed behavior:

- bounded file-touching flows can now produce file-work records automatically
  when workflow context is available
- this includes:
  - direct runtime dispatch paths
  - MCP RPC `tools/call` paths
- explicit `file_work_record` remains available and valid

Covered bounded tool set:

- `edit_file`
- `copy_path`
- `move_path`
- `delete_path`
- `save_file`
- `restore_file_from_disk`

This means the runtime can preserve useful file-work context without requiring
every bounded file touch to be followed by a second manual reminder step.

## 3.4 Policy clarification for `tool_not_found`

The release also clarified an important AI-agent-facing policy.

A bounded attempted file-touch can still be useful recovery context even when the
visible `ctxledger` MCP tool surface returns:

- `tool_not_found`

The intended reading is:

- if the attempted action exposed bounded file-touch metadata
- and that metadata helps later resumability, recovery, or explanation
- the attempted file-touch is still worth recording as a durable support trail

This avoids throwing away useful operational context just because the visible
surface did not directly implement the attempted tool.

## 3.5 Live proof for automatic file-work recording

The small-auth stack was used to validate the automatic file-work path in a live
running environment.

Closed result:

- a real bounded file-touching `tools/call` flow produced a durable automatic
  file-work record in PostgreSQL
- live data showed:
  - `recording_mode = automatic_from_http_runtime`
  - `source_tool_name = edit_file`
  - concrete `file_path`
  - concrete `file_operation`
  - preserved `purpose`

This matters because the release is not only implementation-complete in tests;
it is also evidenced in the intended running stack.

## 3.6 `memory-stats` observability repair

Live validation exposed a separate observability bug:

- `memory-stats` failed because PostgreSQL memory summary repositories did not
  support the expected count path

Closed result:

- PostgreSQL memory summary repositories now support counting for:
  - `memory_summaries`
  - `memory_summary_memberships`
- live `memory-stats` now succeeds
- canonical summary volume and derived-graph readiness are visible through the
  CLI without runtime failure

## 4. Derived-memory observability closeout

## 4.1 Problem that was closed

Before this slice, operator-facing derived-memory panels could collapse into an
ambiguous `No data` reading.

That was a poor operator signal because it failed to distinguish between:

- normal absence
- canonical-only state
- degraded derived-layer state
- unknown observability conditions

## 4.2 Explicit state model introduced

The `1.0.1` slice introduced explicit derived-memory observability fields across
SQL, Grafana, and CLI surfaces.

Added fields include:

- `derived_memory_item_count`
- `derived_memory_item_state`
- `derived_memory_item_reason`
- `derived_memory_graph_status`
- `latest_derived_memory_item_created_at`

The intended operator reading now distinguishes:

- `ready`
- `not_materialized`
- `canonical_only`
- `degraded`
- `unknown`

## 4.3 Surfaces aligned

The explicit derived-memory reading is now surfaced in:

- observability SQL
- Grafana memory overview
- `memory-stats`
- `stats`

This alignment matters because operators should not have to switch mental models
between dashboard and CLI surfaces.

## 4.4 Current live interpretation example

The current bounded live reading can now express a state like:

- `derived_memory_graph_status = graph_ready`
- `derived_memory_item_state = canonical_only`

This means:

- the derived graph layer is readable
- canonical summary state exists
- derived memory items themselves are not materialized

That is much more useful than a generic `No data` panel.

## 5. Validation summary

The bounded `1.0.1` slice was validated through both automated tests and live
small-auth runtime checks.

Representative validation outcomes:

- targeted Python test suites remained green through the hardening work
- the small-auth Docker stack stayed healthy after restart-oriented validation
- live AGE readiness returned:
  - `graph_ready`
- live CLI observability returned canonical summary and derived-state signals
- live automatic file-work recording was observed in PostgreSQL with bounded
  source metadata

This release should therefore be read as **implemented, validated, and
operationally evidenced** for its intended bounded scope.

---

## 6. Canonical boundary after `1.0.1`

The stabilization work in this release does **not** change the canonical system
of record.

After `1.0.1`:

- PostgreSQL remains canonical
- workflow and checkpoint truth remains canonical-first
- canonical summaries and summary memberships remain relational-first
- graph-backed structures remain derived support layers
- file-work metadata is durable and important, but still subordinate to canonical
  workflow truth
- observability fields are operator signals, not a replacement truth system

---

## 7. What `1.0.1` does not try to solve

This closeout does not claim that `1.0.1` fully solves every future derived-layer
question.

It does **not** close:

- stronger truth-sourcing for derived-memory observability beyond the current
  bounded readiness-backed posture
- a final long-term materialization policy for derived memory items
- all future operator verification ergonomics
- broader `1.1.0` follow-on observability and derivation themes

Those belong to the next milestone planning wave.

---

## 8. Release interpretation

The correct release interpretation is:

- `1.0.0` remains the Azure milestone
- `1.0.1` closes the follow-on reliability and observability hardening slice
- the release leaves the system easier to restart, easier to interpret, and
  easier to recover from bounded file-touching agent work
- the release turns several ambiguous operational readings into explicit,
  explainable operator signals
- the release does not introduce a new truth model or a new major architectural
  layer

In short:

`1.0.1` should be read as the post-Azure stabilization patch where the current
`ctxledger` operating model became significantly more trustworthy in live
restart, file-work, and observability behavior.

---

## 9. Follow-on milestone boundary

Future work should now be framed as `1.1.0` planning rather than `1.0.1` release
debt.

That includes themes such as:

- tighter truth-sourcing for derived-memory observability
- richer linkage between derived state and live AGE readiness
- possible formal materialization strategy for derived memory items
- stronger release-facing operator verification procedures

`1.0.1` itself is complete.
