# v0.1.0 Acceptance Evidence Matrix

## 1. Purpose

This document records the current acceptance evidence for `v0.1.0` by mapping each practical requirement to:

- implementation evidence
- public surface evidence
- test evidence
- documentation evidence
- current assessment

It is intended to support closeout review for `v0.1.0` and reduce ambiguity about what is already implemented versus what is only planned.

---

## 2. Scope

This matrix focuses on the acceptance criteria implied by:

- `docs/imple_plan_0.1.0.md`
- `docs/imple_plan_review_0.1.0.md`

Especially the `v0.1.0` practical surface around:

- the primary HTTP MCP transport
- workflow tools
- workflow resources
- PostgreSQL-backed durability
- startup/runtime visibility
- documentation
- basic test evidence

---

## 3. Status Legend

- **Confirmed**  
  Strong visible repository evidence exists across implementation and tests.

- **Partially Confirmed**  
  There is meaningful evidence, but some public-surface or closeout detail is still incomplete.

- **Open**  
  The repository still needs more evidence or additional work before the criterion should be treated as closed.

## 6. Remaining Closeout Gaps

The main remaining `v0.1.0` gaps are now more about closeout precision than about whether a minimal HTTP MCP endpoint exists at all.

The most important unresolved area is the **acceptance boundary of the primary HTTP MCP transport**:

1. **A minimal HTTP MCP endpoint at `/mcp` is now evidenced**
   - visible repository evidence now proves HTTP MCP `initialize`
   - visible repository evidence now proves HTTP MCP `tools/list`
   - visible repository evidence now proves HTTP MCP `tools/call`

2. **Acceptance evidence is still too stdio-heavy**
   - some broader MCP protocol evidence is still concentrated on stdio
   - `v0.1.0` acceptance must remain centered on remote HTTP MCP behavior

3. **Public surface correction is still required**
   - keep a concise canonical matrix of MCP vs non-MCP HTTP routes
   - reduce drift across docs
   - avoid treating workflow/debug HTTP routes as if they proved more MCP transport scope than is actually tested

4. **Optional stronger end-to-end evidence**
   - add tighter Docker/bootstrap verification if needed
   - decide whether required-tool and resource proof must be expanded beyond the currently evidenced minimal HTTP MCP path

---

## 7. Closeout Assessment

Based on the currently visible implementation, tests, and docs:

- core workflow-domain behavior: **strongly evidenced**
- workflow resume naming alignment: **confirmed**
- stdio MCP tool schema discoverability: **confirmed on stdio**
- PostgreSQL durability baseline: **confirmed**
- runtime/debug visibility: **confirmed**
- basic workflow-path tests: **confirmed**
- primary HTTP MCP endpoint at `/mcp`: **proven for the minimal path**

### Practical conclusion

`v0.1.0` should **not yet** be treated as fully closeable, but it should also no longer be framed as blocked by the absence of an HTTP MCP endpoint.

The main remaining work is now best understood as:

- deciding whether the currently proven minimal HTTP MCP protocol surface at `/mcp` is sufficient for `v0.1.0`
- re-centering acceptance evidence on HTTP rather than stdio
- correcting documentation and release framing so they reflect current evidence without overstating readiness
