# v0.1.0 Acceptance Evidence Matrix

## 1. Purpose

This document records the current acceptance evidence for `v0.1.0` by mapping each practical requirement to:

- implementation evidence
- public surface evidence
- test evidence
- documentation evidence
- current assessment

It is intended to support closeout review for `v0.1.0` and reduce ambiguity about what is already implemented versus what is only planned.

---

## 2. Scope

This matrix focuses on the acceptance criteria implied by:

- `docs/imple_plan_0.1.0.md`
- `docs/imple_plan_review_0.1.0.md`

Especially the `v0.1.0` practical surface around:

- the primary HTTP MCP transport
- workflow tools
- workflow resources
- PostgreSQL-backed durability
- startup/runtime visibility
- documentation
- basic test evidence

---

## 3. Status Legend

- **Confirmed**  
  Strong visible repository evidence exists across implementation and tests.

- **Partially Confirmed**  
  There is meaningful evidence, but some public-surface or closeout detail is still incomplete.

- **Open**  
  The repository still needs more evidence or additional work before the criterion should be treated as closed.

## 6. Remaining Closeout Gaps

The main remaining `v0.1.0` gaps are now mostly about final closeout precision rather than whether a practical remote HTTP MCP path exists.

The most important remaining areas are:

1. **The remote HTTP MCP path is now evidenced beyond the original minimal boundary**
   - visible repository evidence proves HTTP MCP `initialize`
   - visible repository evidence proves HTTP MCP `tools/list`
   - visible repository evidence proves HTTP MCP `tools/call`
   - visible repository evidence now also proves HTTP MCP `resources/list`
   - visible repository evidence now also proves HTTP MCP `resources/read`

2. **Docker end-to-end evidence now exists**
   - a FastAPI/`uvicorn` runtime is started through Docker Compose
   - a minimum MCP client reaches `/mcp` on the live server
   - basic MCP lifecycle and workflow tool flows are validated against a real PostgreSQL-backed deployment

3. **The main remaining closeout work is documentation precision**
   - keep a concise canonical matrix of MCP vs non-MCP HTTP routes
   - reduce drift across docs
   - ensure release framing reflects the now-confirmed FastAPI/Docker HTTP evidence rather than older stdio-heavy framing

4. **Optional stronger final proof**
   - add any additional acceptance-specific scenarios if release signoff requires them
   - extend resource/history verification if a broader release checklist demands it

- polishing acceptance documentation so it reflects the confirmed FastAPI/Docker remote MCP evidence

- keeping release framing precise so it reflects current evidence without overstating broader protocol completeness
